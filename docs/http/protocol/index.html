Abschnitt 1 - Das Protokoll

Zuerst einmal müssen wir uns über 2 Begriffe klar sein: Client und Server. Das sind die beiden Seiten, welche per HTTP-Protokoll miteinander reden.

Der Client ist etwa der Web Browser, in dem eine Webseite geöffnet wird.

Der Server ist der Computer oder Rechner, auf dem die Webseite gehostet wird.

Wenn wir das Internet nutzen, stellen wir als Client ständig Anfragen (Request) an verschiedene Server, die uns den gewünschten Inhalt wie Informationen, Bilder oder Videos zurückliefern (Response).

Request und Response​

HTTP basiert auf dem zustandslosen Austausch von Nachrichten. Was bedeutet das?

Stateless bedeutet zustandslos. Fragen wir eine Videodatei beim Server an, gibt er uns diese zurück. Fragen wir nun erneut nach der Videodatei an, weiß der Server nichts mehr davon, ob oder wann wir bereits einmal angefragt haben. Alle Informationen, die der Server braucht, um unsere Anfrage zu erfüllen, müssen in jeder Anfrage enthalten sein.

Für einige Anwendungsfälle wie Webanwendungen mit Login ist es notwendig, dass der Server sich Informationen über uns merkt. In einem späteren Abschnitt werden wir sehen, wie das möglich ist.

Wie genau findet der Austausch von Nachrichten statt? HTTP definiert eine sogenannte Anfrage (Request) und eine Antwort (Response). Auf jeden Request folgt genau 1 Response.

Request​

Im Wesentlichen besteht ein HTTP-Request aus Text in der folgenden Form:

Wir erkennen daran 3 wesentliche Teile:

  • Statuszeile
  • Header
  • Body

Statuszeile

GET https://example.com/ HTTP/1.1

In der Statuszeile stehen 3 Informationen. Als erstes kommt das sogenannte HTTP-Verb, welches angibt, um welche Art von Anfrage es sich handelt. Danach steht die URL mit der Adresse, an welche die Anfrage gesendet wird. Als letztes steht die Version des HTTP-Protokolls. Mittlerweile gibt es auch HTTP 2/3, dies werden wir uns später kurz anschauen.

Das HTTP-Verb sollten wir uns nochmal genauer anschauen. HTTP definiert verschiedene Aktionen oder Arten von Anfragen mit verschiedener Bedeutung, zum Beispiel:

  • GET - Hole ein Dokument oder eine Datei ab. Ein GET-Request ist das, was immer ausgeführt wird, wenn wir im Browser eine Webseite aufrufen.
  • POST - Sende Daten an den Server. Oft verwendet bei Formularen, um die Formulardaten abzusenden.
  • PUT - Ersetze die Zieladresse mit dem gesendeten Inhalt. Oft verwendet, um etwa eine Datei hochzuladen.
  • DELETE - Lösche die Datei an der Zieladresse.
  • HEAD - Erhalte Meta-Informationen zur Seite, mache sonst aber nicht weiter.

Die Bedeutung dieser HTTP-Verben ist nur schwach definiert. Theoretisch kann jeder Server selber definieren, was genau bei welchem HTTP-Verb passiert. Es ist aber hilfreich, sich an die übliche Bedeutung zu halten. Speziell sollte ein GET-Request immer nur etwas zurückliefern und nie ein Änderung wie Anlegen oder Löschen von Daten auf dem Server bewirken.

Header

Als nächstes folgend die sogenannten HTTP-Header. Ein HTTP-Header besteht aus einem Namen und einem Wert. Der Nutzer einer Webseite sieht diese Header nie direkt. Header dienen dazu, Metadaten zum Request oder zum Response bereitzustellen. Beispielsweise kann im Request der Browser dem Server mitteilen, in welcher bevorzugten Sprache der Server die angefragte Webseite zurückliefern soll:

Accept-Language: de-DE

Einige bekannte Header werden wir uns in Kürze anschauen. Wichtig ist noch, dass die Namen von Header case-insensitive sind, Groß- und Kleinschreibung also keine Rolle spielt. Egal, ob wir den Header Accept-Language, accept-language oder AcCePt-LaNgUaGe nennen, das Resultat ist das gleiche.

Body

Der Body enthält die eigentlichen Daten der Anfrage.

Rufen wir eine Webseite nur auf, ist der Body leer. Senden wir ein Formular ab , stehen im Body die Werte der einzelnen Formularfelder.

Der HTTP-Standard selber erlaubt im Body beliebige binäre Daten. Über den Header Content-Type wird angegeben, welches Format die Daten im Body haben.

Ein typisches Format ist application/json, wenn also im Body ein JSON-Wert (meist Array oder Objekt) steht. Das wird besonders gerne von sogenannten REST-API verwendet, wenn also 2 Programme miteinander über HTTP kommunizieren. Bei Single-Page-Apps besteht die Webseite beispielsweise oft nur aus JavaScript, welches dann per REST-API Daten vom Server anfragt und dem Nutzer darstellt.

Zwei weitere typische Formate sind application/x-www-form-urlencoded und multipart/form-data.

Nehmen wir hierzu nochmal ein einfaches HTML-Formular als Beispiel. Wir setzen noch das Attribut method="post", damit der Browser die Daten im Body sendet:

<form action="https:/example.com/post" method="post" enctype="application/x-www-form-urlencoded">
  <label for="email">E-Mail</label>
  <input id="email" name="email" type="email">
  <button name="btnSubmit" type="submit">Absenden</button>
</form>

Mit dem Attribut enctype="application/x-www-form-urlencoded" weisen wir den Browser an, die Daten beim Absenden im sogenannten Format x-www-form-urlencoded zu übertragen. Vereinfacht gesprochen bedeutet das, die Daten werden nach dem Schema name1=value1&name2=value2&... übertragen. Dabei ist name der Name des Formular-Elements und value dessen Wert.

info

Falls Sonderzeichen im Namen oder Wert auftauchen, werden diese URL-kodiert. Das URL-Encoding schauen wir uns im nächsten Abschnitt genauer an.

Klicken wir nun auf den Submit-Button, sendet der Browser etwa folgendes Request an den Server:

POST https://example.com/example.com/post HTTP/1.1
accept-language: de-DE
user-agent: Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/133.0.0.0 Safari/537.36
content-length: 41
content-type: application/x-www-form-urlencoded

email=test%40example.com&btnSubmit=

Wir erkennen, dass im Body die Daten der Formularfelder stellen. Bei Dateien ist dies Format etwas ungünstig. Ändern wir nun den enctype auf enctype="multipart/form-data" und senden erneut ab, sehen wir folgendes Request:

POST https://example.com/example.com/post HTTP/1.1
accept-language: de-DE
user-agent: Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/133.0.0.0 Safari/537.36
content-length: 254
content-type: multipart/form-data; boundary=----WebKitFormBoundaryCNomoyMHu4lWFbBZ

------WebKitFormBoundaryCNomoyMHu4lWFbBZ
Content-Disposition: form-data; name="email"
test@example.com

------WebKitFormBoundaryCNomoyMHu4lWFbBZ
Content-Disposition: form-data; name="btnSubmit"

------WebKitFormBoundaryCNomoyMHu4lWFbBZ--

Response​

Die Response stellt die Antwort vom Server dar. Dessen Aufbau ist ähnlich zum Request:

Wir erkennen wieder die gleichen Bestandteile:

  • Statuszeile
  • Header
  • Body

Header und Body sind gleich zum Request.

In der Statuszeile steht beim Response kein HTTP-Verb mehr, stattdessen finden wir einen sogenannten Status-Code. Eine Liste aller Status-Codes findet sich bei MDN web docs - Status.

Der Status-Code gibt einen groben Aufschluss darüber, ob das Request erfolgreich war und welche Art von Fehler aufgetreten ist, wenn nicht. Ein Status-Code besteht immer aus 3 Ziffern, wobei die 1. Ziffer die Kategorie des Status-Codes darstellt.

  • 2xx (200 - 299) - Status-Codes 2xx bedeuten Erfolg. Der häufigste ist 200 ("OK"), der zurückgeliefert wird, wenn alles OK war.
  • 3xx (300 - 399) - Status-Codes 3xx bedeuten Weiterleitung. Rufen wir etwa eine Webseite auf und der Server antwortet mit 302 ("Found"), leitet der Browser an die URL weiter, die im Header Location zu finden ist.
  • 4xx (400 - 499) - Status-Codes 4xx stehen für clientseitige Fehler. Ein clientseitiger Fehler ist ein solcher, wo der Client verantwortlich ist, also beispielsweise die falsche URL aufgerufen hat (404 - Not Found), nicht die richtigen Werte mit übertragen hat (400 - Bad Request) oder sich nicht angemeldet hat (401 - Unauthorized).
  • 5xx (500 - 599) - Status-Codes 5xx stehen für serverseitige Fehler, also solche, wo der Server für den Fehler verantwortlich ist. Das kann etwa passieren, wenn wir als Programmierer einen Bug im Server-Code eingebaut haben (500 - Internal Server Error) oder wenn ein Administrator den Server gerade wartet (503 - Service Unavailable).

Übliche HTTP-Header​

Es gibt inzwischen viele verschiedene HTTP-Header für verschiedene Einsatzzwecke. Bei MDN web docs findet sich eine Liste vieler Header. An dieser Stelle wollen wir nur wenige wichtige erwähnen, die uns begegnen werden.

Accept​

Header: Accept

Gibt beim Request an, mit welche Arten von Dateien der Client umgehen kann. Ein typischer Wert beim Aufruf einer Webseite könnte etwa sein: text/html,application/xhtml+xml,application/xml,image/avif,image/webp,image/png. Dieser Client kann also HTML, XML und Bilder verarbeiten. Bei REST-API wird der Accept-Header zudem verwendet, um die Art von Inhalt zu beeinflussen ( z.B. JSON oder XML).

Accept-Language​

Header: Accept-Language

Die Sprache, in der die Antwort zurückgeliefert werden soll. Das können wir testen, indem wir zum Beispiel per curl die Startseite von MDN web docs aufrufen:

$ curl 'https://developer.mozilla.org/' -H 'accept-language: en-US' -v
Found. Redirecting to /en-US/

$ curl 'https://developer.mozilla.org/' -H 'accept-language: ja-JA' -v
Found. Redirecting to /ja/

Je nach Header leitet MDN entweder auf die englische oder die japanische Sprachversion weiter.

Content-Type​

Header: Content-Type

Gibt den Inhalt des Body an, z.B. Content-Type: text/plain. Der Wert ist eine Media-Type (auch: MIME-Type). Ein Media-Type ist eine standardisiertes Format, um den Inhalt einer Datei anzugeben.

Ein Media-Type folgt dem Format:

primaryType/subType

Der Primary Type gibt die grundlegende Dateiart an, der Sub-Type den konkreten Typ. Folgende wesentliche Primary Type gibt es:

  • text - Textdokumente
  • image - Bilder
  • audio - Ton, Musik
  • video - Videos
  • application - anwendungsspezifische Dateien

Einige konkrete Beispiele für Dateien, denen wir im Web häufig begegnen, sind:

  • text/plain - Reines Textdokument
  • text/html - HTML-Dokument
  • text/css - CSS-Datei
  • text/javascript - JavaScript-Datei
  • image/png - PNG-Bild
  • image/jpeg - JPEG-Bild
  • application/svg+xml - SVG-Bild
  • application/json - JSON-Datei
  • audio/aac - AAC-Audio-Datei
  • audio/ogg - OGG-Audio-Datei
  • video/webm - Webm-Video
  • video/H265 - H.265-kodiertes Video

Weiteres offizielle registrierte Media-Type für bestimmte Dateiformate finden sich auf der Webseite von IANA, der dafür zuständigen Organisation.

Location​

Header: Location

Dieser Header wird bei Weiterleitungen durch den Server verwendet, um den Client auf eine andere Webseite weiterzuleiten.

Erwähnenswert ist noch, dass es verschiedene Arten von Weiterleitungen gibt, abhängig vom Status-Code:

  • 301 (Moved Permanently) - Die Seite ist permanent umgezogen. Der Browser merkt sich dies und fragt beim nächsten Aufruf an die URL nicht mehr beim Server nach, sondern leitet gleich zur neuen URL weiter. Wir sollten diesen Status-Code daher nur mit Vorsicht verwenden, da es schwer rückgängig zu machen ist.
  • 302 (Found) - Temporäre Weiterleitung. Der Browser fragt bei jedem Aufruf erneut beim Server nach, ob die Weiterleitung noch gültig ist. Der häufigste Status-Code für Weiterleitungen.
  • 307 (Temporary Redirect) und 308 (Permanent Redirect) - Analog zu 302 und 301, aber mit dem Unterschied, dass das HTTP-Verb beibehalten wird.

Origin​

Header: Origin

Der Origin-Header gibt die URL der Webseite an, von der aus der Client oder Browser den Request gestellt hat. Wenn wir etwa die Webseite https://example.com/webapp besuchen und dort per JavaScript eine HTTP-Request ausgeführt wird, steht im Origin-Header https://example.com (also ohne den Pfad am Ende).

Dieser Header ist besonders wichtig bei sogenannten Cross-Origin-Requests (CORS). Ein Cross-Origin-Request ist ein HTTP-Request zu einer anderen Webseite. Wenn das JavaScript auf https://example.com/webapp ein HTTP-Request an https://api.restful-api.dev/objects/7 stellt, um von dort Daten abzuholen, handelt es sich um ein CORS-Request.

Wir müssen bei CORS-Requests, die von JavaScript gestartet werden, aufpassen. Aus Sicherheitsgründen blockieren Browsers solche Requests standardmäßig. Erlaubt ist uns solch ein Request nur, wenn der Server, bei dem wir anfragen, dies explizit erlaubt.

Will der Server von https://api.restful-api.dev/objects/7 dies erlauben, muss dieser Server in der HTTP-Response den Header Access-Control-Allow-Origin setzen. Als Wert muss hier der Origin hinterlegt sein, der zugelassen werden soll.

Und hier kommt der Origin-Header aus dem Request ins Spiel. Üblicherweise liest der Server den Origin-Header aus und prüft, ob er diesen zulassen möchte. Wenn ja, nimmt der Server diesen Wert und schreibt ihn in den Access-Control-Allow-Origin-Header der Response.

Zusammengefasst:

  • Request-Header: Origin: https://example.com
  • Response-Header: Access-Control-Allow-Origin: https://example.com

User-Agent​

Header: User-Agent

Im User-Agent sendet der Client mit, um welche Art von Anwendung oder Browser es sich handelt. Daran kann der Server erkennen, ob es sich um Chrome, Firefox, Safari, eine Crawl-Bot einer Suchmaschine etc. handelt.

Auf den User-Agent können wir auch im JavaScript per window.navigator.userAgent zugreifen.

Viele Webbrowser sind inzwischen dazu übergegangen, hier einen generischen String zu hinterlegen, der keinen Rückschluss mehr auf den Browser zulässt, z.B. nutzt der Chrome:

'Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/133.0.0.0 Safari/537.36'

Auf dem Server sollten wir uns nur auf Web-Standards verlassen. Im JavaScript gibt es kaum Gründen, herauszufinden, um welchen Browser es sich handelt. Wollen wir feststellen, ob der Browser ein bestimmtes Feature unterstützt, gibt es hierzu andere Möglichkeiten.

HTTPS / TLS​

Im ursprünglichen HTTP-Standard war nicht vorgesehen, Verbindungen zu verschlüsseln. Senden wir etwa ein Anmeldeformular ab, wird das Passwort im Klartext und unverschlüsselt übertragen. Mit Wachstum des Internets hat sich dies als problematisch erwiesen.

Fast alle Webseiten, die wir heutzutage aufrufen, beginnen mit https://, nicht mit http://. s steht hierbei für secure und bedeutet, dass die Daten über eine verschlüsselte Verbindung übertragen werden. Der Client verschlüsselt die Daten und der Server entschlüsselt die Daten. Niemand dazwischen kann die Daten lesen.

In anderen Worten, die Daten sind beim Übertragen über die Internetverbindung verschlüsselt, daher nennt man dies auch Verbindungsverschlüsselung.

Über HTTPS hinausgehend gibt es noch das Konzept der sogenannten End-To-End-Verschlüsselung.

Nehmen wir als Beispiel eine Chat-Anwendung. 2 Chat-Teilnehmer, nennen wir sie Anna und Boris, wollen miteinander reden. Anna sendet eine Nachricht an den Chat-Server. Boris ist gerade nicht online. Der Chat-Server speichert die Nachricht und wartet, bis Boris wieder online ist. Dann leitet er ihm Annas Nachricht weiter.

Sowohl die Verbindung von Anna zum Chat-Server als auch von Bob zum Chat-Server ist per HTTPS gesichert. Das Problem ist nun, dass der Chat-Server alle Daten im Klartext lesen kann. Falls Emil in der Firma arbeitet, die den Chat-Server betreibt und böse Absichten hat, kann er die Nachrichten von Anna und Boris mitlesen.

Um das zu verhindern, kann End-To-End-Verschlüsselung verwendet werden. Dabei einigen sich Anna und Boris auf ein Geheimnis (Secret). Nun kann Anna ihre Nachrichten selber verschlüsseln und an den Chat-Server senden. Der Chat-Server speichert nur die verschlüsselte Nachricht und leitet diese an Boris weiter, der die Nachricht entschlüsseln kann. Die Nachricht ist also nicht nur während eines HTTP-Request-Response verschlüsselt, sondern auf dem gesamten Weg zwischen beiden Enden der Verbindung (Anna und Boris). Da der Chat-Server kein Kenntnis vom Secret hat, kann der Chat-Server die Nachricht auch nicht entschlüsseln.

HTTP 2/3​

HTTP/2 und HTTP/3 sind Ausbaustufen von HTTP/1.1.

An der grundlegenden Funktionsweise, nämlich dem Austausch von Nachrichten mittels Request und Response ändert sich dabei nichts. Die Verbesserungen und Änderungen sind eher technischer Natur (Transportschicht) und betreffen unter Anderem auch Performance-Probleme in HTTP/1.1.

In der Regel kann ein Server auf HTTP/2 oder HTTP/3 umgeschaltet werden, ohne das eine Webanwendung oder Webseite angepasst werden muss. Dadurch können bereits Geschwindigkeitsvorteile erzielt werden. Durch gezielte Nutzung von neuen Features wie Multiplexing kann die Webseite oder Webanwendung potentiell noch mehr optimiert werden.
