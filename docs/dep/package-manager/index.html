Abschnitt 1 - Package Management

Notwendigkeit von Bibliotheken​

Der Begriff Bibliothek (Library) bezieht auf Software von Drittanbietern. Also hier konkret auf JavaScript-Code, der von anderen Personen geschrieben wurde.

Um nur einige Beispiele zum besseren Verständnis zu nennen:

  • lodash ist eine Bibliothek mit vielen allgemeinen Hilfsfunktionen
  • ibankit zum Validieren von Internationale Bankkontonummern (IBAN).
  • PDF.js von Mozilla erlaubt die Darstellung von PDF-Dokumenten im Browser.
  • Axios unterstützt bei der Kommunikation via HTTP
  • @microsoft/microsoft-graph-client vereinfacht die Arbeit mit der Microsoft Graph API.
  • Bouncy Castle stellt kryptographische Algorithmen (Verschlüsselung) bereit
  • Jest hilft bei der Erstellung automatischer Tests
  • D3 ermöglicht die Visualisierung von Daten in Form interaktiver Diagramme
  • Pixi hilft bei der Erstellung von browserbasierten WebGL-Inhalten und Spielen,

Abhängig von den Lizenzbedingungen können wir diesen Bibliotheken verwenden. Warum sollten wir das tun? Können wir den Code nicht selber schreiben? Es gibt viele gute Gründe

  • Zeitersparnis — Natürlich könnten wir Hilfsfunktionen wie lodash selber schreiben, aber dann erfinden wir das Rad neu. Nutzen wir die Bibliothek, können wir uns stattdessen auf die eigentlichen Business-Anforderungen konzentrieren.
  • Komplexität — Themen wie WebGL oder PDF sind so komplex, dass wir uns lange in die Fachlichkeiten einarbeiten müssten, um den Code selber zu schreiben.
  • Edge Cases — Themen wie Kryptographie erfordern nicht nur Fachwissen, sondern auch gewissenhaftes Testen. Sicherheitslücken entstehen oft in Randfällen (Edge Cases), die wir als Programmierer nicht bedacht haben. Daher stammt die Empfehlung, nie selber sicherheitsrelevante Algorithmen zu schreiben.

Sollten wir also so viele Bibliotheken wie möglich verwenden? Wie gesagt kann der Einsatz sinnvoll sein, andererseits müssen wir aber auch die Nachteile bedenken, die der Einsatz einer Bibliothek hat. Es muss Gründe geben, warum große Projekte wie TypeScript sich entschieden haben, keine Abhängigkeiten, also Zero Dependencies, zu verwenden. Einige Punkte, die wir bei der Verwendung von Bibliotheken bedenken sollten:

  • Wartung — Bei langfristigen Projekten ist es wichtig, dass diese auch noch in vielen Jahren funktionieren. Stellt der Entwickler einer Bibliothek etwa aus privaten Gründen ein, müssen wir einen Ersatz hierfür finden. Das wird sehr gut illustriert im Web-Comic XKCD 2347
  • Sicherheit — Code von Drittanbietern, den wir nicht kontrollieren, stellt ein potentielles Sicherheitsproblem dar. Bei sogenannten Supply-Chain-Angriffen wird Schadcode in einer Bibliothek injiziert, die von vielen eingesetzt wird.
  • Bugs — Fehler in Software sind nahezu unvermeidbar. Fehler, die wir selber durch unseren Code behoben haben, können wir schnell und einfach beheben. Falls aber eine von uns verwendete Bibliothek einen Fehler hat, ist es in der Regel schwieriger und benötigt mehr Zeit, um diesen Fehler zu beheben.
  • Konflikte zwischen Abhängigkeiten — Bibliotheken können selber wieder andere Bibliotheken verwenden. Wenn wir viele Bibliotheken verwenden, können so Konflikte entstehen. Z.B. wenn Bibliothek A die Bibliothek C in Version 1 und Bibliothek B die Bibliothek C in Version 2 benötigt.
  • Inkompatibilitäten der Lizenz

Verwendung von Bibliotheken​

Unter Abwägung der Vor- und Nachteile haben wir uns nun entschieden, die Bibliothek lodash zu verwenden. Wir tun wir das?

Bei JavaScript ist die einfachste Möglichkeit, das JavaScript über einen <script>-Tag einzubinden. Bei lodash konkret könnte das wie folgt aussehen:

In Sandbox öffnen
lib-lodash
loading...

Das funktioniert recht gut bei eigenständigen Bibliotheken, für die es eine einzelne JavaScript-Datei gibt. Oft haben Bibliotheken aber wiederum andere Bibliotheken als Abhängigkeit (Dependency). In dem Fall müssen wir diese das JavaScript dieser Abhängigkeiten auch einbinden und dabei die Reihenfolge beachten. Es ist nicht schwer, sich vorzustellen, dass der Aufwand hierbei wesentlich steigt, je mehr Bibliothek wir verwenden: 10 Bibliotheken, von denen jede Bibliothek von 3 anderen Bibliotheken abhängt und jede dieser 3 Bibliotheken wiederum von 2 weiteren Bibliotheken etc.

warnung

cdn.jsdelivr.net ist ein sogenanntes Content Distribution Network (CDN).

CDN vereinfachen die Einbettung von Ressourcen und erhöhen die Ladegeschwindigkeit durch Server in verschiedenen Ländern. Um die Ressource zu laden, muss der Browser aber gezwungenermaßen ein HTTP-Request an das CDN stellen. Gesetze wie die Datenschutz-Grundverordnung erfordern, dass das CDN den entsprechenden datenschutzrechtlichen Anforderungen gerecht wird. Ob und in welcher Weise etwa die Übertragung der IP-Adresse durch bloßes Laden einer Ressource vom CDN per GET-Request eine Verletzung des Datenschutzes darstellt, ist umstritten.

Aus diesem Grund kann es sinnvoll sein, Ressourcen soweit möglich von unseren eigenem Servern auszuliefern, wo auch die Webseite oder Webapp läuft.

Alternativ können wir auch die JavaScript-Ressourcen herunterladen und lokal in unserem Webseiten-Projekt hinterlegen. Diese Dateien werden dann wie unser selbstgeschriebener JavaScript-Code mit ausgeliefert. Hier gibt es aber mindestens 2 Probleme.

Zum Einen ist das Update der Bibliothek schwieriger. Fast alle Bibliothek haben eine Version. Behebt der Autor der Bibliothek Fehler behoben oder fügt neue Funktionalitäten hinzu, wird eine neue Version der Bibliothek veröffentlicht. Dabei können sich auch die Abhängigkeiten ändern, welche die Bibliothek benötigt. Wir müssen beim Update dann nicht nur die JavaScript-Datei neu herunterladen und in unserem Projekt einfügen, sondern auch alle Abhängigkeit prüfen und gegebenenfalls aktualisieren.

Zum Anderen besteht die Gefahr, dass wir in Versuchung geraten und im Quellcode der Bibliothek Änderungen vornehmen. Das ist ungünstig, weil damit das Verhalten der Bibliotheken geändert und wird und etwa nicht mehr mit der Dokumentation der Bibliothek übereinstimmt. Bei einem Update der Bibliothek auf eine neue Version haben wir Probleme, wenn wir vergessen, die Änderung erneut vorzunehmen.

Package Manager​

Ein Package Manager löst diese Probleme bei der Verwendung von Abhängigkeiten. Er kümmert sich um das Ermitteln aller benötigten Abhängigkeiten und um das Herunterladen der Abhängigkeiten.

Viele Programmiersprachen haben inzwischen einen solchen Package Manager eingeführt. Beispielsweise heißen diese:

  • Composer bei php
  • Maven oder Gradle bei Java
  • RubyGems bei Ruby
  • NuGet bei C#
  • Cargo bei Rust

Allen gemeinsam ist, dass wir in einer Manifest-Datei definieren, welche Bibliotheken wir gerne in welcher Version nutzen möchten. Der Package Manager kümmert sich dann um den Rest.

Für JavaScript gibt es mehrere verschiedene Package Manager. Der ursprüngliche und immer noch weiter verbreitete ist NPM. Andere populäre Package Manager sind Yarn und pnpm. Diese unterscheiden sich in Details, das meiste in diesem Kapitel ist auf alle Package Manager anwendbar. Wir beschränken wir uns hier auf einen groben Überblick von NPM.

info

Historisch bedingt bezieht sich der Begriff NPM sowohl auf den Package Manager selber als auch auf den Online-Service (die sogenannte NPM-Registry), welcher die Abhängigkeiten und Bibliotheken bereitstellt. Dieser Online-Service wird auch von den anderen Package Managern Yarn und pnpm verwendet.

NPM​

warnung

Zum Ausprobieren von NPM muss dieses erst einmal installiert werden. Für Linux empfiehlt sich die Installation per nvm (Node Version Manager). Für Windows kann es von der offiziellen Webseite heruntergeladen werden.

Wie genau funktioniert das Management von Packages (Bibliotheken) nun mit NPM?

Herzstück eines JavaScript-Projekts mit NPM ist die Datei package.json. Das ist eine JSON-Datei und enthält grundlegende Informationen über unser JavaScript-Projekt:

{
  "name": "my-project",
  "version": "0.0.1",
  "description": "This project is awesome",
  "main": "index.js",
  "scripts": [],
  "dependencies": {}
}
info

Über den Befehl npm init können wir auch eine solche package.json-Datei erzeugen lassen.

In der Eigenschaft dependencies stehen alle Bibliotheken, die wir verwenden möchten. Um also etwa lodash zu verwenden:

{
  "name": "my-project",
  "version": "0.0.1",
  "description": "This project is awesome",
  "main": "index.js",
  "scripts": [],
  "dependencies": {
    "lodash": "~4.17.21"
  }
}

Mit "lodash": "~4.17.21" sagen wir NPM, dass wir gerne lodash in Version 4.17.21 verwenden möchten. Die Tilde ~ bedeutet, dass bei Konflikten auch Versionen mit einer anderen Versionsnummer an letzter Stelle (z.B. 4.17.24) verwendet werden dürfen.

Nun müssen wir nur noch den folgenden Befehl ausführen:

npm install

NPM lädt lodash und alle eventuell benötigten Abhängigkeiten herunter. Wo genau diese auf der Festplatte gespeichert werden, hängt vom Package Manager ab. NPM speichert diese im Unterordner node_modules des aktuellen Ordners.

Jetzt können wir lodash verwenden. Zum Beispiel können wir die Datei index.js erstellen und darin folgendes schreiben:

import _ from "lodash";

console.log("Was lodash loaded - ", typeof _ === "function");
console.log(_.partition([1, 2, 3, 4], n => n % 2));

Zum Ausführen verwenden wir folgenden Befehl:

node index.js

Haben wir alles richtig gemacht, wird lodash geladen und kann verwendet werden.

hinweis

Statt manuell den Namen und die Version der Abhängigkeit in die package.json zu schreiben, können wir auch den Befehl npm install lodash verwenden. Dabei wird lodash in der package.json mit der aktuellen Version eingetragen und im Anschluss gleich heruntergeladen.

Node.js​

Einen Moment — haben wir eben JavaScript außerhalb des Browsers ausgeführt? Wie ist das möglich?

Die Antwort lautet Node.js. Node.js ist ein Programm, mit dem JavaScript ohne Browser auf dem Rechner ausgeführt werden kann. Technisch wird dabei die V8-Engine verwendet, die auch von Chromium-basierten Browsern zur Ausführung von JavaScript eingesetzt wird.

Node.js als Server​

Node.js hat mehrere Anwendungsfälle. Eine Anwendungsfall ist die Nutzung als Webserver. Auch ganz ohne Abhängigkeiten und Drittbibliotheken können wir mit Node.js einen Server starten, da Node.js bereits entsprechende APIs dafür mitbringt. Wir werden uns in dieser Vorlesung nicht mit der serverseitigen Programmierung beschäftigen, daher an dieser Stelle nur eine kurzes Beispiel:

import http from "node:http";

const port = 8080;
const host = "127.0.0.1";

const server = http.createServer((request, response) => {
  const url = new URL(request.url, `http://${host}`);
  const x = parseFloat(url.searchParams.get("x"));
  const y = parseFloat(url.searchParams.get("y"));
  response.statusCode = 200;
  response.setHeader("Content-Type", "text/plain");
  response.end([
    `You made a ${request.method} request to ${url}`,
    `${x} + ${y} = ${x + y}`,
  ].join("\n"));
});

server.listen(port, host, () => {
  console.log(`Server running at http://${host}:${port}/`);
});

Wenn wir obige Datei als server.js speichern, können wir mittels node server.js die JavaScript-Datei ausführen und den Server starten. Dieser Code gibt die URL aus, an die das Request gestellt wurde. Zudem wird die Summe der Werte aus den URL-Parameter x und y zurückgegeben. Rufen wir etwa den Server über die URL http://127.0.0.1:8080/foo?x=5&y=3 auf, erhalten wir folgende Antwort:

You made a GET request to http://127.0.0.1/foo?x=5&y=3
5 + 3 = 8

Node.js als Build-System​

Ein anderer Anwendungsfall ist die Nutzung als Build-System. Zum Beispiel ist der Package Manager NPM selber ist in JavaScript geschrieben und erfordert Node.js. Auch TypeScript ist in JavaScript geschrieben. Mittels Node.js können wir TypeScript unseren Code überprüfen und in normales JavaScript umwandeln lassen.

Allgemein können Node.js verwenden, um aus unserem Quellcode mittels verschiedener Tools zu verarbeiten und die finalen Ressourcen für unserer Webseite oder Webapp zu erstellen.

Im nächsten Abschnitt schauen wir uns einige typische Kategorien von Tools an, die häufig verwendet werden.

Weitere JavaScript-Runtimes​

Node.js war die erste JavaScript-Runtime für den Server, die große Bekanntheit erfuhr. Der Vollständigkeit halber sei noch angemerkt, dass aktuell auch noch andere weitere Plattformen für die Ausführung von JavaScript erstellt wurden:

  • Deno — Der Name entstand durch Vertauschung der Silben von No-de. Im Wesentlichen verhält sich Deno ähnlich wie Node.js, hat aber einige eingebauten Funktionalitäten wie die native Unterstützung zur Ausführung von TypeScript (*) und verbesserte eingebaute APIs.
  • Bun — Während Node.js und Deno die V8-Engine zur Ausführung von JavaScript verwenden, setzt Bun auf JavaScriptCore, der JavaScript-Engine von Webkit (Safari-Browser). Es behauptet von sich, sowohl bei der Verwaltung von Abhängigkeiten als auch bei der Ausführung schneller zu sein.

(*) Mittlerweile ermöglicht auch Node.js die direkte Ausführung von TypeScript.

Dev-Dependencies​

Wir hatten gesehen, dass wir mittels dependencies in der package.json eine Abhängigkeit angeben konnten, die installiert werden soll. Erinnern wir uns:

{
  "name": "my-project",
  "version": "0.0.1",
  "description": "This project is awesome",
  "main": "index.js",
  "scripts": [],
  "dependencies": {
    "lodash": "~4.17.21"
  }
}

Die dependencies sind für Abhängigkeiten gedacht, die wir tatsächlich zur Laufzeit benötigen, also etwa wenn ein Nutzer unsere Webapp im Browser lädt.

Es gibt aber auch Abhängigkeiten, die wir nur für einen Build-Schritt brauchen, etwa zum Prüfen unseres TypeScript auf Typenfehler; oder zum Ausführungen von automatisierten Software-Tests. Solche Abhängigkeiten sollten wir als sogenannte devDependencies deklarieren:

{
  "name": "my-project",
  "version": "0.0.1",
  "description": "This project is awesome",
  "main": "index.js",
  "scripts": [],
  "dependencies": {
    "lodash": "~4.17.21"
  },
  "dependencies": {
    "typescript": "~5.8.3",
    "jest": "~29.7.0"
  }
}

Falls wir eine Bibliothek schreiben, die wir auf NPM veröffentlichen, verhindert das, dass NPM automatisch diese devDependencies mit herunterlädt und installiert, wenn jemand unsere Bibliothek installiert.

Semver​

Wir haben gesehen, dass jedes Bibliothek eine Version hat. Diese Versionen folgen dem Schema x.y.z. Prinzipiell könnte die Version nach Belieben und Ermessen der einzelnen Entwickler festgelegt werden. In der Praxis hat sich aber ein Schema für die Festlegung von Versionen bewährt.

Dieses Schema nenn sich Semantic Versioning, oft abgekürzt zu semver. Bei Semver hat jeder Teil der Version eine bestimmte Bedeutung:

MajorVersion.MinorVersion.PatchVersion

In der Version 3.1.9 beispielsweise ist 3 die Major-Version, 1 die Minor-Version und 9 die Patch-Version. Wenn wir eine neue Version von unserer Software veröffentlichen, müssen wir je nach Art der vorgenommenen Änderungen die Version anpassen:

  • Major-Version — Die Major-Version muss erhöht werden, wenn eine neue Version der Bibliothek inkompatibel mit der bisherigen Version ist. Wenn wir beispielsweise eine Funktion entfernen, müssen wir die Major-Version hochsetzen.
  • Minor-Version — Die Minor-Version steht für neue Funktionalitäten, die mit der bisherigen Version kompatibel sind. Fügen wir eine neue Option hinzu, die optional ist, brauchen wir nur die Minor-Version zu erhöhen.
  • Patch-Version — Die Patch-Version erhöhen wir, wenn wir nur Fehler in der aktuellen Version beheben.

Dieses Vorgehen zur Benamung von Versionen hat mehrere Vorteile.

Es erleichtert uns zum Einen die Arbeit, wenn wir die Abhängigkeiten unserer Anwendung aktualisieren. Nur anhand der Versionsnummer erhalten wir bereits eine grobe Vorstellung davon, welche Änderungen an einer Abhängigkeit vorgenommen wurde. Setzen wir etwa aktuell die Abhängigkeit immer in Version 2.3.1 ein und ist ein Update auf 2.3.9 verfügbar, können wir uns bereits recht sicher sein, dass wir dieses Update problemfrei durchführen können. Falls es bereits die Version 2.6.2 gibt, sollten wir kurz über den Changelog (Beschreibung der Änderungen) schauen, fühlen uns aber immer noch recht sicher. Falls aber schon die Version 4.8.0 verfügbar ist, werden wir das Update nicht sofort durchführen und erst genau prüfen, was bei dem Update zu beachten ist.

Ein zweiter Vorteil ergibt sich bei Verwendung von einem Package Manager wie NPM. Um das zu illustrieren, nehmen wir an, unsere Webapp hätte folgende Abhängigkeiten in den folgenden Versionen:

Zum Verständnis: Unserer Webapp benötigt zum Beispiel die Abhängigkeit axios in Version 1.2.0. axios benötigt die Abhängigkeit form-data in Version 2.1.1.

In diesem konkreten Fall gibt es ein Problem: Die Abhängigkeit btoa wird dreimal in 3 verschiedenen Versionen benötigt, form-data in 2 verschiedenen Versionen. Der Package Manager NPM ist prinzipiell in der Lage, mit so einem Fall umzugehen, indem die Abhängigkeiten in mehreren Versionen heruntergeladen und entsprechend zugeordnet werden. Dennoch ist das problematisch. Die Gesamtgröße unserer Anwendung wird dadurch erhöht, zudem kann es zu subtilen Problemen kommen, wenn eine Bibliothek globale Variablen verwendet und sich darauf verlässt, nur einmal geladen zu werden.

Und das bringt uns zurück zu Semver. Wenn wir davon ausgehen, dass sich die Abhängigkeiten an Semver halten, dann brauchen wir etwa die Abhängigkeit btoa nur einmal herunterladen in Version 2.5.0. Es unterscheiden sich nur die Minor-Versionen. 2.5.0 hat möglicherweise neue Funktionalitäten, ist aber abwärtskompatibel mit 2.1.0 und 2.3.4. Das gleich gilt für form-data. Modul-Systeme​

Zum Schuss dieses Abschnitts noch einige Worte zu Modulsystemen in JavaScript.

Historisch hatte JavaScript kein Modulsystem. Falls eine JavaScript-Datei eine andere Datei oder Bibliothek als Voraussetzung (Abhängigkeit) hatte, mussten wir die entsprechende Datei manuell einbinden und sicherstellen, dass diese in der richtigen Reihenfolge geladen werden.

<script src="https://cdnjs.cloudflare.com/ajax/libs/autonumeric/1.9.46/autoNumeric.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/jquery/3.4.0/jquery.min.js"></script>
<script src="../js/my-webapp.js"></script>

Dieses Vorgehen hat sich als impraktikabel erwiesen, wenn die Anzahl der Abhängigkeiten steigt. Besser wäre es, wenn jede JavaScript-Datei sagen könnte, welche Abhängigkeiten sie hat. Dann können diese automatisch geprüft und aufgelöst werden. Jede Datei nennt sich dann Modul.

In der Vergangenheit gab es mehrere Versuche von Drittanbietern zu Modulsystemen. 2 bekannte Modulsysteme sind CommonJS und AMD.

CommonJS etwa wurde von Node.js entwickelt und sieht in der Praxis etwa wie folgt aus:

const { get } = require("axios");

module.exports.getSampleData = async (count) => {
  const response = await get(`/data?count=${count}`);
  return response.data;
}

Mit require sagen wir, dass wir eine Abhängigkeit benötigen, mit module.exports stellen wir anderen Dateien Funktionen bereit.

CommonJS ist aber kein offizieller Standard und hat verschiedene Probleme. Beispielsweise ist es sehr schwierig für Tools, genau herauszufinden, welche Funktionen eine Datei bereitstellt. Etwa erlaubt CommonJS, den Namen eines Exports dynamisch zu berechnen:

module.export[Math.random()] = "Hallo";

Seit 2015 gibt es mit EcmaScript2015 ein offizielles Modulsystem namens ESM — EcmaScript Modules. ESM wird inzwischen sowohl vom Browser als auch von Node.js unterstützt. Unser Beispiel von eben sieht mit ESM wie folgt aus:

import { get } from "axios";

export async function getSampleData(count) {
  const response = await get(`/data?count=${count}`);
  return response.data;
}

Mit import importieren wir Funktionen von anderen Modulen und Dateien, mit export stellen wir Funktionen anderen Dateien und Modulen bereit.
