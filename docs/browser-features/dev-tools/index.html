Abschnitt 2 - Weitere Dev-Tools-Funktionen

In diesem Abschnitt betrachten wir noch einige weitere nützliche Tools, die uns bei der Entwicklung unterstützen können. Dev-Tools stellen inzwischen viele Funktionen für uns Entwickler bereit, wir stellen hier nur einige ausgewählte vor.

Print-Simulation​

Wir hatten im Kapitel zu CSS kennengelernt, dass wir mittels Media Queries CSS erstellen konnten, was nur beim Drucken einer Seite angewandt wird. Um etwa den Text beim Drucken schwarz zu machen:

@media print {
  * {
    color: black !important;
  }
}

Ausprobieren können wir das, in dem wir unsere Webseite öffnen und den Druckdialog des Browsers öffnen. Bei komplexen CSS kann es aber schwierig sein, Fehler zu finden. Dev-Tools stellen uns eine einfach Möglichkeit bereit, den Media-Type testweise auf print zu stellen.

Wo genau diese Funktion zu finden ist, hängt vom Browser ab. Bei Chromium-basierten Browser finden wir diese Funktion im Tab Rendering. Bei Firefox gibt es einen Button zum Aktivieren der Print-Vorschau im Elements-Tab.

Responsiveness​

Mit der Verbreitung von Smartphones, Tablets und anderen Geräten müssen wir bei der Webentwicklung beachten, dass unserer Webseite auch auf Bildschirmen mit verschiedener Auflösung und Größe verwendbar ist.

Wir sollten immer die Webseite auf verschiedenen Geräten testen und prüfen. Während der Entwicklung kann das aber umständlich sein. Dev-Tools stellen uns Möglichkeiten bereit, schnell einen Überblick darüber zu erhalten, wie unsere Webseite auf mobilen Geräte in etwa aussehen wird. In Chromium-basierten Browsern finden wir links oben einen Umschalter, um den Device-Emulations-Modus zu aktivieren.

Ist der Device-Modus aktiviert, können wir dynamisch die Bildschirmgröße ändern und beobachten, wie sich die Webseite verhält. Zudem ändert sich der Maus-Cursor zu einem kleine Kreis. Technisch werden jetzt Pointer Events statt Mouse Events ausgelöst. Hiermit können wir schnell feststellen, ob wir alle entsprechenden Event Listener richtig registriert haben.

Performance​

Stellen wir Performance-Probleme auf unserer Webseite fest, bietet der Performance-Tab uns Möglichkeiten, um die Ursache dieser Probleme herauszufinden.

Wir sehen im Performance-Tab bereits einige relevante Informationen zu unserer Webseite, etwa den Largest Contentful Paint (die Zeit, bis der Hauptteil der Webseite geladen und dem Nutzer angezeigt wurde).

Um das Verhalten auf Geräten mit eingeschränkter Leistung oder Netzwerk zu beobachten, stehen uns (oben rechts) auch Möglichkeiten zur Verfügung, um die CPU und das Netzwerk zu limitieren.

Schränken wir die CPU ein, benötigt JavaScript länger zu Ausführung. Interagieren wir jetzt mit unserer Webseite, können wir überprüfen, ob die Webseite immer noch angenehm zu bedienen ist.

Schränken wir das Netzwerk ein, simulieren wir das Verhalten bei schlechter Internetverbindung. Das ist besonders dann nützlich, wenn unsere Webseite großen Gebrauch von HTTP-Requests zum Nachladen von Inhalten macht. Hiermit können Wir etwa prüfen, ob der Nutzer etwa mit Ladeanimation oder ähnlichen Feedback versorgt wird, um ein angenehmes Erlebnis auf der Webseite zu haben.

Weiterhin erlauben uns die Dev-Tools noch, aufzuzeichnen, was während der Interaktion mit der Webseite passiert, also z.B. welches JavaScript ausgeführt oder welche Netzwerk-Requests ausgelöst werden. Über den Record-Buttons links oben können wir die Aufzeichnung starten, dann mit der Webseite interagieren und schließlich die Aufzeichnung wieder beenden.

Events​

Im Elements-Tab der Dev-Tools gibt es rechts auch den Punkt Event Listeners:

Hier können wir einsehen, für welche Events per JavaScript ein Event-Listener registriert wurde. Zudem wird auch angezeigt, welche JavaScript-Funktion als Callback registriert wurde und ausgeführt werden wird, wenn das Event eintritt. Klicken wir auf den Link, können wir auch direkt zu der entsprechenden im Quellcode navigieren. In dem konkreten Fallen sehen wir, dass ein Event-Listener auf das click-Event registriert ist, also wenn der Nutzer auf den Button klickt.

DOM-Breakpoints​

Der Elements-Tab hat auch noch eine weitere nützliche Funktion. Mit einem Rechtsklick können wir einen sogenannten DOM-Breakpoint am Element aktivieren.

In dem Sandbox-Beispiel für Media-Devices hatten wir einen Button, um die Anzeige der Videodaten der Webcam wieder zu stoppen. Technisch war die Anzeige über ein <video>-Element umgesetzt. Beim Klick auf den Stop-Button wird das <video> aus dem DOM entfernt.
