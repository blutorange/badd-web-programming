Abschnitt 10 - Typisierung

Bisher konnten wir in JavaScript-Dateien alles schreiben, was wir wollten, und vom Browser ausführen lassen. Manchmal sind bei diesem Versuch Scherben geflogen und wir haben Fehler vom Browser erhalten.

Wäre es nicht aber schön, wenn die IDE oder ein Checker schon vor Ausführung des Codes auf potentielle Fehler prüfen könnte? Auch wenn dabei nicht alle Fehler und Bugs auffalen, ist es hilfreich, die Menge an Fehlern zu reduzieren. Zudem besteht die Gefahr, dass wir beim Ausführen von Code Fehler übersehen, die in Code-Teilen stecken, die nur selten ausgeführt werden (Fehlerbehandlung, Edge Cases).

Dies ist die Idee von Typen-Checkern und einem Typen-System.

Mit Typ ist die Art von Wert gemeint, die eine Variable besitzt. Als Programmierer legen wir für jede Variable fest, welchen Typ diese hat. Der Type-Checker kann dann prüfen, ob diese Typen durch das gesamte Programm hinweg konsistent sind. Sind sie es nicht, wird ein Fehler gemeldet. Andernfalls ist gewährleistet, dass die Variablen auch zur Laufzeit den entsprechenden Typ haben. Typen-Systeme für JavaScript​

Mit zunehmender Komplexität von JavaScript-Code und Webanwendungen wurden verschiedene Typensysteme für JavaScript entwickelt:

  • TypeScript unter der Federführung von Microsoft
  • Flow, unter der Federführung von Facebook

Im grundlegenden Gebrauch sind beide Typensystem sehr ähnlich. Der folgende Code ist sowohl gültiges TypeScript als auch gültiges Flow:

append-to-list
loading...

Inzwischen hat TypeScript am meisten Verwendung gefunden, daher wollen wir uns in diesem Abschnitt TypeScript etwas genauer anschauen.

Design-Goals​

TypeScript hat sich verschiedene Ziele gesteckt, die es erfüllen möchte. Wir wollen uns zwei wichtige davon anschauen.

Nur Type-Checker​

TypeScript versteht sich nur als Type-Checker für JavaScript. Ziel ist nicht, neuen Syntax oder Features hinzuzufügen, die nicht mit normalen JavaScript möglich sind.

An allen Variablen, Funktionsparametern, Rückgabewerten und Klassenfeldern können wird in TypeScript nach einem Doppelpunkt : hinterlegen, von welchem Typ der Wert sein soll:

const age: number = 12;
function sum(x: number, y: number): number { ... }
class User {
  name: string;
  age: number
}

Diese Typen nach dem Doppelpunkt nennt man Typen-Annotationen.

Anders formuliert ist Ziel von TypeScript, dass wir diese Typen-Annotationen entfernen können und dann bereits gültiges JavaScript erhalten, welches zum Beispiel im Browser ausgeführt werden kann.

note

Historisch besitzt TypeScript einige Features, welche diesem Design-Goal widersprechen. Im Wesentlichen sind dies Enum, Namespace und Parameter Properties. Zur Wahrung der Abwärtskompatibilität werden diese Features noc hunterstützt. Falls möglich, würde ich aber empfehlen, diese nicht mehr zu verwenden. Inzwischen gibt es in den Compiler-Options auch das Flag erasableSyntaxOnly, um solche Konstrukte zu verbieten.

Balance zwischen Korrektheit und Benutzbarkeit​

Ein Typen-System nennt man sound, wenn der Type-Checker beweisbar garantieren kann, dass zur Laufzeit alle Variablen vom deklarierten Typ sind (und somit keine Fehler aufgrund falscher Typen auftreten können).

Nun ist JavaScript eine sehr dynamische Sprache. Ein Typen-System, was diese Garantie liefern kann, würde auch schwerwiegende Einschränkungen mit sich zienen, welchen Code wir schreiben können. Damit die Typenprüfung erfolgreich ist müssten wir unseren Code länger und weniger elegant schreiben, was die Benutzbarkeit der Sprache für uns Programmierer reduziert.

Aus diesem Grund hat sich TypeScript als Ziel gesetzt, eine Balance zu finden zwischen Korrektheit der Typen und Bedienbarkeit der Sprache für Programmierer.

Ein Beispiel, wo der Type-Checker keinen Fehler erkannt, zur Lauftzeit aber ein Typenfehler auftritt:

unsound-opt-prop
loading...

In den letzten Jahren hat sich TypeScript in Richtung mehr Korrektheit entwickelt. Es gibt verschiedene Optionen, die man setzen kann, um die Korrektheit zu erhöhen, beispielsweise:

  • strictFunctionTypes
    • Besser Prüfung von Funktionsparametern und Rückgabewerten, wenn Funktionen etwa als Argumente an andere Funktionen übergeben werden.
  • noUncheckedIndexedAccess
    • Davon ausgehen, dass beim Zugriff auf den Index eines Arrays (array[4]) immer undefined erhalten werden könnte, wenn das Array nicht so lang ist.
  • useUnknownInCatchVariables
    • Bei try { ... } catch (e) {} davn ausgehen, dass e jeder beliebige Wert sein könnte.
note

Durch Vermeidung einiger Features wie optionalen Eigenschaften, optionalen Funktionsparametern und Klassen sowie Nutzung von Features wie readonly kann bei Bedarf die Typen-Sicherheit weiter erhöht werden.

Alternative typisierte Sprachen​

Wie erwähnt erschwert die Programmiersprache JavaScript es, typensicheren Code zu schreiben. Es gibt einige alternative Programmiersprachen, welche dieses Problem umgehen. Dies erreichen sie dadurch, dass sie kein gültiges JavaScript mehr sind, sondern eine eigene Programmiersprache, die aber durch einen Build-Prozess in JavaScript überführt werden kann.

Beispiele für solche Sprachen sind:

  • Dart
    • Eine beinahe typensichere Sprache, welche sowohl in nativen Binärcode für die Ausführung auf dem Server sowie in JavaScript für die Ausführung im Browser überführt werden kann.
  • PureScript
    • Eine von Haskell inspirierte, stark typisierte funktionale Programmiersprache, die in JavaScript übersetzt wird.
  • Elm
    • Ebenso eine funktionale Sprache, mit einem Schwerpunkt auf dem Erstellen von Webanwendungen.

Bei letzteren beiden ist ein Hauptproblem aus meiner Sicht aktuell die mangelnde Verbreitung und damit einhergehend die fehlende Dokumentation, Community und 3rd-Party-Bibliotheken.

Transpilation​

Wenn wir TypeScript schreiben, kann dieser Code nicht direkt im Browser ausgeführt werden. Es müssen wenigstens die Typen-Annotationen entfernt werden. Dieser Prozess wird manchmal als Transpilieren bezeichnet -- abgeleitet von Kompilieren.

Die Notwendigkeit zur Transpilation stellt einen Nachteil von TypeScript dar, da (1) so ein zusätzlicher Schritt notwendig ist, bis unser Code lauffähig ist. Zudem erschwert es (2) auch die Fehlersuche (Debugging), da der im Browser ausgeführte nicht der gleiche ist, den wir geschrieben haben.

Das erstere Problem kann umgangen werden, indem Typen-Annotationen nicht direkt im Code, sondern in den Doc-Kommentaren eingefügt werden. TypeScript unterstützt solche Typen-Annotationen in Kommentaren. Der Nachteil dabei ist, dass der Code so etwas aufgebläht wird und Typen-Annotationen nicht mehr ganz so einfach zu schreiben sind.

Aufgrund der großen Beliebtheit von Typen in JavaScript gibt es ein TC39-Proposal, Raum für Typen-Annotationen in der Syntax von JavaScript zu schaffen: Type Annotations. Diese müssen als Art Kommentar verstanden werden -- JavaScript selber prüft diese nicht. Dafür wären immer noch 3rd-Party-Tools notwendig. Vorteil wäre aber eine einheitliche Sytnax und die Möglichkeit, typenannotiertes JavaScript ohne Transformationschritt direkt auszuführen.

info

Einige Programme, die JavaScript ausführen können, unterstützen inzwischen auch direkt TypeScript. Beispielsweise unterstützt node.JS die Ausführung von TypeScript über Flags.

Basis-Typen​

TypeScript unterstützt die Basic-Typen von JavaScript:

const a: undefined = undefined;
const b: boolean = false;
const c: number = 0;
const d: string = "Hallo TypeScript";
const e: string[] = ["Typen", "wie", "dich", "finde", "ich", "toll"];

Für Objekte wird in der Regel erst einmal definiert, welche Eigenschaften das Objekt überhaupt besitzen kann:

interface User {
  name: string;
  age: number;
  active: boolean;
}

const user: User = { name: "Estelle", age: 21, active: true };

In JavaScript ist es nicht unüblich, verschiedene Typen gleichzeitig zu erlauben. Hierzu können wir ein sogenannten Union-Type verwenden:

union-type
loading...

Diese Union-Type können wir auch verwenden, um eine Aufzählung möglicher Werte abzubilden. Dies ist oft besser als eine Enum. Wenn wir beispielsweise eine Funktion schreiben wollen, die eine Nachricht auf der Konsole ausgibt, und das entweder als Information, Warnung, oder Fehler:

string-constant-union
loading...

Darüber hinaus stellt TypeScript noch viele weitere fortgeschrittene Typen zur Verfügung. Teilweise wurden diese entwickelt, um bestehende JavaScript-Bibliotheken abzubilden, die nicht in TypeScript geschrieben wurden und Gebrauch von den dynamischen Eigenschaften der Sprache JavaScript machen.

Beim Schreiben von neuen TypeScript-Code sollten wir mit, und nicht gegen, den Type-Checker arbeiten. Das bedeutet auch, unseren Code so zu strukturieren, dass der Computer diesen einfach prüfen und möglichst viele Probleme frühzeitig erkennen kann. Viele der fortgeschrittenen Typen von TypeScript sollten daher mit Bedacht verwendet und wenn möglich vermieden werden.

Nutzung von JavaScript-Bibliotheken​

Nicht alle 3rd-Party-Bibliotheken sind in TypeScript geschrieben. Beispielsweise ist jQuery eine bekannte, viel verwendete Bibliothek, welche rein in JavaScript geschrieben ist.

Theoretisch können wir solche Bibliotheken in TypeScript verwenden. Es fehlt dann aber jegliche Typen-Sicherheit, Typen-Prüfung und Autocomplete. Um auch in JavaScript geschriebene Bibliotheken komfortabel verwenden zu können, gibt es sogenannten Typen-Deklarationsdateien. Ähnlich zu C-Header-Dateien enthalten diese keinen Code, sondern nur die Typen für die Funktionen, Variablen und Klassen, welche durch die JavaScript-Bibliothek bereitgestellt werden. Damit können wir die Bibliothek so nutzen, als ob diese in TypeScript geschrieben wäre.

Natürlich benötigen wir dafür erst einmal eine solche Deklarationsdatei. Wir könnten diese selber schreiben, dass ist aber zeitaufwendig. Aufgrund der Beliebtheit und weiten Verbreitung von TypeScript existiert eine zentrale Sammelstelle für solche Deklarationsdateien, die für viele JavaScript-Bibliotheken entsprechend Deklarationen enthält:

DefinitelyTyped @ https://github.com/DefinitelyTyped/DefinitelyTyped

Für die eben erwähnte Bibliothek jQuery finden sich die Deklarationsdateien etwa unter folgender URL:

https://github.com/DefinitelyTyped/DefinitelyTyped/tree/master/types/jquery
