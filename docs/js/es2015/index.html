Abschnitt 2 - ES2015

Bisher haben wir die grundlegenden Funktionalitäten von JavaScript besprochen. Mitte 2015 wurde die 6. Version von EcmaScript veröffentlicht, welche daher auch als ES2015 oder ES6 bezeichnet wird.

Die Revision hat der Programmiersprache wesentliche neue Möglichkeiten hinzugefügt, die den Programmieralltag erleichtern (wenn man sie denn erst einmal gelernt hat!)

Im Folgenden wollen wir uns die wesentlichen Neuerungen anschauen. Anmerkung: Einige Teile der Neuerungen wurden erst in den nachfolgenden Jahren veröffentlicht, wir werden diese ab in diesem Abschnitt mit behandeln.

Template-Literale​

Es ist nicht selten, dass wir Strings erzeugen müssen, deren Inhalt sich teilweise dynamisch aus Variablen ergibt. Eine Möglichkeit hierzu ist die Stringverkettung mittels dem +-Operator:

function add(x, y) {
  console.log(x + " plus " + y + " ergibt " + (x+y));
}

Zum Beispiel gibt ein Aufruf von add(1, 2) auf der Konsole 1 plus 2 ergibt 3 aus. Mit Template-Literalen können wir solche Strings etwas übersichtlicher erstellen:

function add(x, y) {
  console.log(`${x} plus ${y} ergibt ${x+y}`);
}

Mittels ${...} können wir also innerhalb des Strings Ausdrücke mit Variablen einfügen.

Asynchrone Operationen​

Eins der wichtigsten Konzepte für die moderne Webentwicklung sind sogenannte Promises (Versprechen).

Um zu verstehen, warum diese eingeführt wurden, müssen wir einen Schritt zurücktreten und uns anschauen, wie asynchrone Operationen bisher in JavaScript behandelt wurden.

Callbacks​

Nehmen wir an, wir haben ein Formular mit einem Eingabefeld für eine Postleitzahl. Nun wollen wir, dass bei der Eingabe nach möglichen zugehörigen Orten gesucht wird. Da wir selber keine Liste von Postleitzahlen und Orten pflegen möchten, müssen wir bei einem externen Dienst anfragen. An dieser Stellen gehen wir davon aus, dass wir eine Funktion findCityForZipCode haben. Als ersten Argument nimmt diese Funktion die Postleitzahl.

Wir erhalten wir nun Zugriff auf den gefundenen Ort? Die erste Idee hierzu wäre vielleicht folgende:

const city = findCityForZipCode("01309");

Was ist aber das Problem dabei? Im Allgemeinen wird der Dienst eine Weile benötigen, bis der Ort gefunden ist. Damit die Funktion den Ort direkt als Rückgabewert zurückliefern kann, muss das Programm warten, bis die Suche abgeschlossen ist.

Das bedeutet aber, dass in der Zwischenzeit nichts anderes getan werden kann. Im Bezug auf Webseiten würde das dazu führen, dass die Webseite in dieser Zeit eingefroren wirkt nicht mehr benutzt werden kann.

Besser wäre es daher, der Funktion nur zu sagen, was getan werden soll, sobald der Ort gefunden wurde. Das können wir mit einer sogenannten Callback-Funktion erreichen:

findCityForZipCode("01309", (city, error) => {
  if (error) return;
  console.log(city);
});

Nun kann das Programm in der Zwischenzeit weiter ausgeführt werden. Die Callback-Funktion wird nur gemerkt und später ausgeführt. Dies nennt man eine asynchrone Operation, da zwischen dem Start der Suche und dem Ende eine bestimmte Zeitspanne liegt, in der anderer Code ausgeführt werden kann.

Was passiert, wenn wir nun weitere asynchrone Operationen ausführen wollen, die vom aktuellen Ergebnis abhängen? Beispielsweise, wenn wir nach eine Liste von Restaurant in dem Ort suchen wollen?

findCityForZipCode("01309", (city, error) => {
  if (error) return;
  findMostFamousRestaurantForCity(city, (restaurant, error) => {
    if (error) return;
    findOpenHoursForRestaurant(restaurant, (hours, error) => {
      if (error) return;
      findUserFreeTimeForHours(hours, (freeTime, error) => {
        if (error) return;
        console.log(`Restaurant visit: ${freeTime}`);
      });
    });
  });
});

Wir erkennen unschwer, dass der Code immer geschachtelter und schwerer zu lesen wird. Auch macht es uns die Aufgabe nicht einfacher, den Code in kleinere Bestandteile aufzutrennen. Für dieses Problem hört man manchmal auch den Begriff callback hell.

Promise​

Der erste Schritt, damit wir wieder Herr über die Lage werden, besteht in der Einführung des Promise-Objektes.

info

Das Konzept Promise gibt es auch in anderen Programmiersprachen. In C# etwa heißt es Task, in Java CompletionStage.

Ein Promise ("Versprechen") kapselt einen Wert, der ist in Zukunft zur Verfügung stehen wird. Konkret kann ein Promise in 3 verschiedenen Zuständen sein:

  • pending - Der Wert wird noch ermittelt.
  • settled - Die Ermittlung des Werts wurde abgeschlossen.
    • resolved - Der Wert wurde erfolgreich ermittelt.
    • rejects - Es gab einen Fehler bei der Ermittlung des Werts.

Bei unserem Beispiel mit der Ermittlung des Orts sieht das dann so aus:

const promise = findCityForZipCode("01309");

Wichtig hierbei ist, dass die Funktion findCityForZipCode sofort ein Promise zurückliefert und weiterer Code ausgeführt werden kann, während die Ermittlung im Gange ist.

Die Frage ist nun, wie greifen wir auf den Wert zu, sobald dieser zur Verfügung steht? Das Promise stellt hierzu eine Methode bereit:

promise.then(city => console.log(city));

Im ersten Moment mag es den Anschein haben, dass wir nun wieder bei Callbacks sind und in das gleiche Problem geraten. Dies ist nicht der Fall -- der Vorteil von Promises ist, dass nach und nach immer wieder die then-Methode gerufen werden kann, ohne schachteln zu müssen:

promise
  .then(city => findMostFamousRestaurantForCity(city))
  .then(restaurant => findOpenHoursForRestaurant(restaurant))
  .then(hours => findUserFreeTimeForHours(hours))
  .then(freeTime => console.log(`Restaurant visit: ${freeTime}`));

Um Fehler zu behandeln, können wir die Methode catch verwenden:

promise 
  .then(city => findMostFamousRestaurantForCity(city))
  .then(restaurant => findOpenHoursForRestaurant(restaurant))
  .then(hours => findUserFreeTimeForHours(hours))
  .then(freeTime => console.log(`Restaurant visit: ${freeTime}`))
  .catch(error => console.error("Could not find free time", error));

Beachte: Die catch-Methode ist nur einmal am Ende notwendig! Wenn eine der Suchoperationen fehlschlägt, werden nachfolgende Suchoperationen abgebrochen und direkt zum catch gesprungen.

Hier ist ein interaktives Code-Beispiel dazu, wobei jede Suchoperation mit einer Wahrscheinlichkeit von 10% fehlschlägt. Mehrfach ausführen, um verschiedene Ergebnisse zu sehen!

Oft werden wir selber keine Promises erstellen, sondern mit Funktionen arbeiten, die Promises zurückliefern. Falls wir einmal aber doch selber Promises, erstellen müssen, gibt es dazu 2 Möglichkeiten.

Ist bereits der Wert bekannt, können wir ein neues Promise im Status resolved erststellen mittels Promise.resolve und Promise.reject:

Promise.resolve("Success");
Promise.reject("Error");

Dies hatten wir auch im oberen Beispiel zur Simulation der Suchfunktionen verwenden.

Async / await​

Promises erleichtern und zwar die asynchrone Programmierung, sind aufgrund der vielen Funktionen aber immer noch nicht das Traumziel. Wäre es nicht schön, wenn wir wieder Code so schreiben könnten, wie wir es ganz am Anfang geplant hatten?

const city = findCityForZipCode("01309");

Dies ist möglich, mit einer kleinen Anpassung! Um die Arbeit mit Promises zu vereinfachen, gibt es die Schlüsselwörter async und await. Zuerst markieren wir eine Funktion mit async. Das sorgt dafür, dass die Funktion immer ein Promise zurückliefert.

Arbeiten mit null und undefined​

In Sprachen, wo es einen speziellen Wert gibt, wenn kein anderer Wert verfügbar ist, muss auf diesen Fall geprüft werden, sonst gibt es einen Fehler. In JavaScript resultiert folgender Code etwa in einem TypeError:

Wenn wir uns nicht sicher sein können, dass ein Wert nie undefined oder null sein kann, müssen wir darauf prüfen:

function startsWith(value, prefix) {
  value = value !== undefined && value != null ? value : "";
  prefix = prefix !== undefined && prefix != null ? prefix : "";
  return value.substring(0, prefix.length) === prefix;
}

Dies ist aber sehr umständlich. In JavaScript gibt es 2 verschiedene Operatoren, um die Arbeit mit undefined und null zu vereinfachen. Gemein ist ihnen, dass sie alle ein Fragezeichen (?) enthalten. Das Fragezeichen steht für optional, also dafür, dass nur möglicherweise ein Wert vorhanden ist.

Set und Map​

Listen oder Mengen von Elementen sind ein grundlegendes Toolset bei der Programmierung. In der Praxis ist es selten, dass ein konkreter Einzelfall behandelt werden muss. Meist muss ein Programm mit einer Menge von Eingabewerten zurechtkommen

  • Webshop mit Warenkorb mit n Artikeln
  • MMORPG-Videospiel mit n Spielern und m Gegnern
  • Formularanwendung mit n Eingabefeldern

Wir kennen bereits Arrays, um Elemente in einer bestimmten Reihenfolge zu speichern. Mit Set und Map stehen noch zwei weitere mächtige Datenstrukturen bereit, um mit Elementen zu arbeiten.

Set

ESM​

Selten ist es so, dass wir Programme schreiben, die auf wenige Hunderte Zeilen Code und in eine Datei passen. Der Übersichtlichkeit halber ist es sinnvoll, Code entsprechend seiner Funktionalität in mehrere Dateien aufzutrennen.

Es stellt sich dann allerdings die Frage, wie eine Datei Code aus einer anderen Datei verwenden kann.

Hierzu stellt die Programmiersprache ESM (EcmaScript-Module) bereit.

Jede Datei ist ein Modul mit einem eigenen Scope. Das bedeutet, alle Variablen und Funktionen sind erst einmal nur in dieser Datei und nirgends anders verfügbar.

Falls das Modul nun bestimmte Funktionalitäten anderen Modulen bereitstellen möchte, muss es dies explizit durch ein export-Statement tun. Das folgende ist ein Auszug aus der Bibliothek esprima, eine in JavaScript geschriebene Bibliothek zum Parsen von JavaScript-Code.

// scanner.js
function hexValue() { /* ... */ }

function octalValue() { /* ... */ }

export class Scanner {
  // ...
}

hexValue und octalValue sind Hilfsfunktionen, welche nur innerhalb des Moduls scanner.js bereitstehen.

Die Klasse Scanner wird exportiert und kann dann in anderen Modulen mittels einem import-Statement importiert werden.

// parser.js
import { Scanner } from "./scanner.js";

export class Parser {
  constructor() {
    this.scanner = new Scanner(code, this.errorHandler);
  }
}

Wie wir sehen, exportiert das Modul parser.js ebenfalls wieder eine Klasse Parser, die wiederum in einem anderen Modul verwendet werden kann. Auf diese Weise können wir komplexe Programme in logische Bestandteile zerlegen.

In Browsern werden solche ESM-Module bereit unterstützt. Hierzu muss der Typ eines eingebundenen Scripts auf module gesetzt werden:

<script type="module" src="https://unpkg.com/lodash-es@4.17.21/throttle.js"></script>

Vorteile sind etwa

  • Einzelne Dateien sind individuell cachebar
  • Es müssen nur die Dateien geladen werden, die wirklich benötigt werden

Ein wesentlicher Nachteil ist, dass für jede Datei eine separate Anfrage an den Server gestellt werden muss. Die Verwendung von HTTP/2 bzw HTTP/3 kann diesen Nachteil aber abmildern.
