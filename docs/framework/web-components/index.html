Abschnitt 5 - Standardisierung (Native Komponenten)

Was ist den meisten UI-Bibliotheken und UI-Frameworks gemeinsam, die wir bisher gesehen haben? Alle basieren auf Komponenten, welcher wiederum zu größeren Komponenten zusammengesetzt werden und so eine größeren Webseite oder Webapp bilden.

Bibliotheken und Frameworks sind keine schlechte Idee, bergen aber immer die Gefahr, dass die Bibliothek oder das Framework irgendwann einmal nicht mehr weiterentwickelt wird. Falls wir auf längere Zeit entwickeln, sollten wir uns auch einmal die Frage nach Standardisierung stellen. Wäre es nicht schön, wenn es in einem Standard wie der DOM API eine Möglichkeit gäbe, Komponenten zu erstellen?

Diese Möglichkeit gibt es nun schon seit einiger Zeit und nennt sich Custom Elements oder auch Web Components.

Überblick Custom Elements​

Registrierung​

Ein Custom Elements ist eine HTML-Element wie div oder button, dass wir selber geschrieben haben. Hier ist ein ganz einfaches Beispiel für ein <hello-world>-Element, welches nur den Text Hallo Welt! ausgibt:

Wir sehen, im HTML können wir das Element wie jedes andere Element über <hello-world> verwenden.

Im JavaScript müssen wir eine Klasse schreiben, welche HTMLElement (aus der DOM API) erbt. Schließlich müssen wir noch unsere Klasse mittels customElements.define bekannt machen. Danach kann unser Element im HTML verwendet werden.

Constructor function​

Die Klasse für das Custom Element, welche von HTMLElement, kann einige spezielle Methoden und Felder besitzen, mit denen wir das Verhalten der Komponente steuern können. Im obigen Beispiel hatten wir schon die Methode connectedCallback gesehen, wo das Element mit dem Text Hallo Welt! erzeugt wurde.

Alle verfügbaren Möglichkeiten finden sich in der Spezifikation. Hier einige der wichtigsten kurz zusammengefasst:

  • constructor — Wenn eine neue Instanz des Elements erstellt werden soll, also z.B. bei jedem Mal, wenn das Element im HTML verwendet wird. Hier können wir Instanzfelder befüllen oder eine Shadow Root (weiter unten erklärt) erzeugen.
  • connectedCallback — Wenn das Element zum DOM hinzugefügt wurde. Hier können wir etwa Event Listener registrieren.
  • connectedCallback — Wenn das Element vom DOM entfernt wurde. Hier können wir etwa Event Listener wieder entfernen.
  • attributeChangedCallback — Wenn ein Attribut geändert wurde, etwa wenn wir in den DevTools des Browser ein Attribut anklicken und dessen Wert ändern oder die DOM-API-Methode setAttribute verwenden. Falls notwendig, müssen wir hier unser internes HTML aktualisieren.
  • observedAttributes — Ein statisches Feld mit den Namen der Attribute, für die attributeChangedCallback aufgerufen werden soll.
class HelloWorld extends HTMLElement {
  static observedAttributes = ["value", "label"];
  
  constructor() {
    console.log("Element wurde neu erstellt");
  }

  connectedCallback() {
    console.log("Element wurde zum DOM hinzugefügt");
  }

  disconnectedCallback() {
    console.log("Element wurde vom DOM entfernt");
  }

  attributeChangedCallback(name, oldValue, newValue) {
    // name ist entweder "value" oder "label", nie etwas anderes,
    // da wir nur diese beiden Attribute in observedAttributes definiert haben.
    console.log(`Attribut ${name} wurde geändert von ${oldValue} zu ${newValue});
  }
}

Ein Beispiel für einen Zähler mit diesen Methoden findet sich weiter unten.

Weitere Möglichkeiten​

Bisher haben wir gesehen, wie Custom Elements definiert und registriert werden können. FÜr die praktische Nutzung stellt der Standard noch weitere Möglichkeiten zur Verfügung, die zwar oft im Zusammenhang mit Custom Elements verwendet werden, aber unabhängig davon sind und prinzipiell auch ohne Custom Elements verwendet werden könnten.

Shadow DOM

Im Kapitel zu HTML hatten wir bereits den Shadow DOM kennengelernt. Erinnern wir uns: der Shadow DOM war eine Möglichkeit, das interne HTML abzutrennen vom Rest der Webseite. In unserem einfachen Beispiel oben hatten wir direkt ein Kindelement erzeugt, in der Praxis ist es aber üblich, das interne HTML in den Shadow DOM einzuhängen.

Den Shadow DOM können wir auf ganz normale Weise mit Custom Elements nutzen, indem wir per attachShadow eine Shadow Root erzeugen und dort dann unsere Kinderelemente einhängen. Das sieht dann wie folgt aus:

Slot

Bisher besteht unser <hello-world> nur aus einem statischen Text. Allgemein können HTML-Elemente auch Kinder haben. Es stellt sich dann die Frage, wo diese Kinderelement in unserem Custom Element dargestellt werden. Slots sind eine Möglichkeit, mit der wir eine oder mehrere Stellen in unserem Custom Element definieren können, wo Nutzer unseres Custom Element beliebige Kinder-Elemente einfügen können.

Als Beispiel passen wir unser <hello-world> so an, dass es 3 Parts gibt, 1 Part für den Inhalt am Anfang, 1 Part für den Inhalt zwischen Hallo und Welt; und einen Part für den Inhalt am Ende:

Part

Der Shadow-DOM kapselt den Inhalt unseres Custom Element von der umgebenden Webseite ab. Dazu gehört auch, dass kein CSS der Webseite auf unser Custom Element Einfluss hat.

Nun kann es aber sein, dass wir wollen, dass Nutzer bestimmte Teile unseres Custom Element per CSS stylen können. Mittels dem HTML-Attribut part können wir Teile unseres Custom Element definieren, auf die Nutzer per CSS zugreifen dürfen. Im CSS können wir den Part dann mit dem Pseudo-Element ::part(name) selektieren.

info

Das CSS der Webseite kann den Inhalt unseres Custom Element indirekt beeinflussen. Genauer gesagt verhält es sich so, dass CSS-Selektoren keine Elemente innerhalb des Shadow-DOM selektieren können. Vererbbare CSS-Attribute finden aber weiterhin Anwendung, auch über Shadow-DOM-Grenzen hinaus. Setzt die Webseite beispielsweise die Schriftgröße auf 20 und die Schriftfarbe auf Rot, dann hat auch der Inhalt unseres Custom Element diese Schriftgröße und Schriftfarbe, sofern wie diese nicht per CSS überschreiben.

Ergebnisse​

Im Folgenden verwenden wir den Begriff Web Components als Sammelbegriff für Custom Elements und weitere APIs wie Slots oder Shadow DOM, die damit im Zusammenhang stehen.

Eine Sache, die uns sofort ins Auge springt, wenn wir unser den Zähler oder den Quellcode des Emoji-Pickers bzw. der TODO-App anschauen: Es ist sehr viel Code.

Im Vergleich zu UI-Bibliotheken und Frameworks sind Web Components bewusst low-level gehalten. Ziel ist, alle Möglichkeiten bereit, die wir brauchen, um Funktionen theoretisch umzusetzen. Aber kein Ziel ist, es uns so einfach wie möglich zu machen.

Auch Reaktivität besteht kaum bis gar nicht. Wir können höchstens (per attributeChangedCallback) auf Änderungen der Attribute reagieren. Aber wenn wir etwa intern ein Datenmodell wie ein Array von TODO-Items haben, stellen uns Web Components keine Möglichkeiten zur Verfügung, bei Änderungen die UI zu aktualisieren. Falls wir reaktive Funktionen brauchen, müssen wir diese entweder selber implementieren oder eine Bibliothek verwenden.

Das Ziel, einen Standard darzustellen, erfüllen Web Components offenbar, denn sie sind ja Bestandteil des HTML Living Standard. Ein Mantra des HTML-Standards ist "Don't break the web". Neue Funktionalitäten oder Erweiterungen bestehender Funktionalitäten werden immer so angelegt, dass alle bestehenden Webseiten immer noch so wie bisher funktionieren. In dieser Hinsicht können wir davon ausgehen, dass Web Components zukunftssicher sind. Web Components können daher ein geeignetes Mittel sein, wenn ein System entwickelt wird, was langfristig betreut werden muss. Auch sind Web Components sinnvoll, wenn es aus externen Gründen nicht möglich oder erwünscht ist, eine Drittbibliothek eines Drittanbieters einzusetzen; oder der Wunsch besteht, Framework-unabhängig zu sein.

Auch bei kleinen, in sich geschlossene Widgets wie dem erwähnte Emoji-Picker oder einem Star Rating können Web Components eine gute Wahl sein. Diese haben in der Regel keine Kinderelemente und können auch in UI-Frameworks wie Svelte oder React verwenden werden (genauso wie native Elemente wie <div>).

Für komplexe, interaktive Anwendungen sind Web Components aber eher nicht das Mittel der Wahl. Allgemein sollten wir immer anhand der Anforderungen prüfen, welche technischen Möglichkeiten in Frage kommen und ob wir Gründe finden, eine Bibliothek oder ein UI-Framework einzusetzen.

Da Web Components wie erwähnt recht viel Code erfordern und nicht sehr einfach anzuprogrammieren sind, gibt es auch Bibliotheken, welche das Ziel haben, die Erstellung von Web Components zu vereinfachen. Beispiele hierfür sind Stencil oder Lit. Diese Bibliotheken bieten Möglichkeiten wie Reaktivität an und erlauben, Komponenten auf andere und einfachere Weisen zu schreiben. Durch einen Build-Prozess wird daraus eine Web Component. Diese Vorgehensweise ermöglicht uns, standardkonforme Komponenten zu erstellen, ohne die Low-Level API des HTML-Standards hierfür direkt nutzen zu müssen.

Auch einige der Bibliotheken, die wir uns bereits angeschaut haben, bieten Möglichkeiten für den Export als Web Components an. Siehe beispielsweise Svelte - custom elements. React selber bietet hier keine Möglichkeit, wir könnten aber selber einen Web-Component-Wrapper schreiben.

Zusammenfassung​

Welche UI-Bibliothek oder Framework sollten wir also verwenden?

Darauf kann keine abschließende Antwort gegeben werden. Es existieren auch noch viele weitere UI-Frameworks wie Angular, Aurelia oder Elm. Wie wir in den Abschnitten dieses Kapitels gesehen haben, lösen diese verschiedene Probleme. Das wirft aber die Frage auf, haben wir bei unserem Anwendungsfall dieses Problem überhaupt?

Wollen wir beispielsweise nur einen Button erstellen, der beim Drücken ein Element ein- und ausblendet, reicht wahrscheinlich die native DOM-API aus, keine Drittbibliotheken benötigt. Haben wir etwa mehr JavaScript, um der Webseite mehr einfache Interaktivität zu verleihen, können wir überlegen, eine Bibliothek wie jQuery oder Cash zu verwenden, um die Arbeit mit dem DOM zu vereinfachen.

Wir die Webseite komplexer oder wollen wir interaktive Apps erstellen, kann es sinnvoll sein, ein Framework mit Reaktivität einzusetzen.

Bei der Auswahl der Bibliothek oder des UI-Frameworks können auch andere Faktoren eine Rolle spielen, wie etwa:

  • Kenntnisstand der Beteiligten mit der Bibliothek oder dem UI-Framework
  • Kompatibilität der Lizenz mit dem Projekt
  • Zuverlässigkeit der Bibliothek (Von dem wird die Bibliothek betreut? Wie bekannt ist diese?)
  • Verfügbarkeit von Drittanbieter- Plugins für die Bibliothek (z.B. Component Libraries)
  • Qualität der Dokumentation

Der Einsatz einer Bibliothek kann uns das Leben erleichtern, birgt aber auch immer Gefahren und Risiken. Wir müssen uns immer die Frage stellen, rechtfertigen die daraus erlangten Vorteile die erhöhte Komplexität und den Overhead?_
