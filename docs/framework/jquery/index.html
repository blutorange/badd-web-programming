Abschnitt 1 - jQuery

Die erste Frage, die wir uns immer stellen sollten, bevor wir eine neue Bibliothek oder ein Framework einsetzen: Warum? Welche Problem wird dadurch gelöst? Wie wird es gelöst?

Wir haben die DOM API kennengelernt, mit der wir prinzipiell alles tun können, was für Widgets und Webapp notwendig ist. Um die Probleme zu sehen, betrachten wir ein einfaches Beispiel für eine Webapp: die TODO-App.

TODO-App​

Die TODO-App ist ein beliebtes Beispiel von UI-Frameworks, um ihre Verwendung zu illustrieren, ein bisschen wie das Hello World-Programm bei der Programmierung. Im Wesentlichen besteht die TODO-App aus:

  • Einer Liste von TODOs, die der Nutzer hinzugefügt hat.
  • Eine Möglichkeit zum Markieren erledigter TODOs
  • Einer Eingabemaske, wo der Nutzer neue TODOs hinzufügen kann.

Zusätzlich können wir noch Features ergänzen, wie etwa einen Filter für offene und erledigte TODOs, einer Möglichkeit zum Löschen von TODOs etc.

An dieser Stelle wollen wir uns orientieren an TodoMVN, einer Sammlung von TODO-Apps in verschiedenen UI-Frameworks.

note

Scherzhaft wird die native Funktionalität, die der Browser bereitgestellt, manchmal auch als Framework Vanilla JS bezeichnet. Das ist eine Anspielung auf die Vielzahl von Bibliotheken und Frameworks, die in JavaScript erstellt wurden (und meist auf *JS enden). Oft versprechen sie viele Features. Wer sich selber an dieser Übung versucht hat, wird bereits einige Probleme erkannt haben. Es gibt mehrere Probleme, von den jQuery einige löst.

Kompatibilität mit älteren Browser​

Wir haben in der TODO-App Gebrauch von verschiedenen DOM-API-Funktionen gemacht, wie classList, closest oder iterierbare NodeList.

Historisch haben nicht alle Browser solche Funktionalitäten überhaupt oder in der gleichen Weise unterstützt.

jQuery war hier eine beliebte Bibliothek, die versprochen hat, dass alle angebotenen Funktionalitäten in allen Browsern auf die gleiche Weise funktionieren.

Inzwischen ist die Standardisierung weiter fortgeschritten, sodass dies kaum noch ein Problem darstellt.

info

@mdn/browser-compat-data stellt umfangreiche Daten bereit, welche Browser welche Features unterstützen. Diese Daten können wir entweder auf den entsprechenden Seiten von MDN web docs oder auf Can I Use einsehen und durchsuchen.

Verbosität​

Die DOM-API ist nicht gerade dafür bekannt, kurz und bündig zu sein. jQuery versucht, dies zu vereinfachen. Beispiele (oben: DOM API, unten: jQuery):

// 1 Element selektieren
document.querySelector(".todo-list");
$(".todo-list");


// Alle Elemente selektieren
document.querySelectorAll(".toggle");
$(".toggle");


// Alle Elemente entfernen
for (const item of document.querySelectorAll(".toggle")) {
  item.parentElement.removeChild(item);
}

$(".toggle").remove();


// Element B nach A einfügen
const a = document.querySelector(".a")!;
const b = document.querySelector(".b")!;
if (b.nextElementSibling) {
    a.insertBefore(b, b.nextSibling);
} else {
    a.appendChild(b);
}

$(".a").after(".b");

Teilweise ist das über die DOM API heutzutage etwas komfortabler möglich, nachdem der Standard weitere Hilfsfunktionen wie after und remove eingeführt hatte.

jQuery​

Wir wollen uns nun noch einen kurzen Überblick über die wesentliche Funktionsweise von jQuery verschaffen. jQuery hat eine sehr gute Dokumentation, die wir unter jQuery API einsehen können.

jQuery lässt sich am besten als Hilfsbibliothek beschreiben, welche neben DOM-API-Funktionalitäten auch noch weitere Hilfen für die tägliche Programmierung bereitstellt. Aufgebaut ist jQuery aus mehreren Teilen, die sich kurz wie folgt zusammenfassen lassen.

DOM-API-Hilfsfunktionen​

Das Kernstück von jQuery sind Hilfsfunktionen rund um die vom Browser bereitgestellte DOM-API. jQuery verwendet üblicherweise die Variable $, weil diese sehr kurz ist.

Selektieren von Elementen

Bevor wir mit Elementen arbeiten können, müssen wir erst einmal welche selektieren. Das ermöglicht jQuery mittels $("cssSelector"). Als Ergebnis erhalten wir eine Liste mit allen Elementen, die auf den CSS-Selektor passen.

Um die Arbeit mit Elementen zu vereinfachen, ist bei jQuery alles eine Liste von Elementen. Auch wenn wir nur ein einzelnes Element selektieren wollen, erhalten wir bei jQuery eine Liste mit einem Element:

In Sandbox öffnen

loading...

Der Vorteil davon ist, dass wir keine Sonderbehandlung für den Fall brauchen, wenn einmal kein Element gefunden wird. Bei der DOM-API kommt im dem Fall null zurück, worauf wir entsprechend reagieren müssen:

// OK, gibt Leerstring zurück
$(".missing").text();

// Schlecht, führt zu Laufzeitfehler
document.querySelector(".missing").textContent;

Arbeiten mit Elementen

Nachdem wir Elemente selektiert haben, können wir damit arbeiten. Hierzu stellt jQuery viele Funktionen bereit, z.B.:

// Neues Element am Anfang vor allen anderen Kindern der 1. <section> einfügen
$("section").first().prepend("<h1>Hallo Welt</h1>");

// Alle .todo-list-item entfernen, deren .toggle angehakt ist (die also erledigt sind) 
$(".todo-list .toggle").filter(":checked").closest(".todo-list-item").remove();

// Neues <input> erstellen, einige Attribute und Klassen setzen, dann
// zum bestehenden <fieldset> hinzufügen
$("<input>")
  .addClass("md-inputfield stylish")
  .prop("required", true)
  .attr("name", "zip code")
  .attr("id", "zip-code")
  .appendTo("fieldset");

Lauschen auf Ereignisse

jQuery vereinfacht auch das Lauschen auf Ereignisse etwas. Erinnern wir an die DOM-API, hier brauchten wir element.addEventListener("event", callback).

In Sandbox öffnen
jquery-events
loading...

Im obigen Beispiel reagieren wir auf Änderungen im Eingabefeld und geben den aktuellen Wert darunter aus.

Was wir an diesem Beispiel auch erkennen: jQuery macht starken Gebrauch vom kontextuellen this-Argument.

Das ist heutzutage aus 2 Gründen problematisch. Zum einen hängt die Bedeutung von this immer von demjenigen ab, der unserer Funktion aufruft. Das ist nicht immer unter unser Kontrolle. Um zu zeigen, dass unser Code korrekt ist, müssen wir Annahmen machen, wie anderer Code (hier: jQuery) unsere Funktion aufruft.

Zum anderen ist this auch einfach inkompatibel mit Arrow-Funktionen (() => {}), da hier this nicht mehr geändert werden kann. Zum Zeitpunkt, wo jQuery erschien, gab es diese noch nicht.

AJAX​

Inzwischen können wir fetch verwenden, um HTTP-Request auszuführen. Als jQuery erschien, gab es fetch noch nicht und wir mussten das sogenannte XMLHttpRequest verwenden. Dies war nicht so komfortabel bedienbar. Grob sah dessen Verwendung etwa wie folgt aus:

const request = new XMLHttpRequest();
request.addEventListener("load", () =>  console.log("response", request.status, request.responseText));
request.addEventListener("error", () => console.error("request failed"));
request.open("GET", "https://api.restful-api.dev/objects?id=3&id=5&id=10");
request.send();

Plugins​

Ein Faktor, der wesentlich zum Erfolg von jQuery beigetragen haben wird, sind Plugins. Als Bibliothek kann jQuery nur eine gewisse Anzahl an Features bereitstellen — jedes Feature muss gewartet werden und erhöht die Größe der Bibliothek.

Fehlt ein Feature, stellt jQuery die Architektur für Plugins bereit, womit externe Entwickler neue Features hinzufügen können. Diese können wir dann verwenden, als ob sie Teil von jQuery wären.

Ergebnisse​

Betrachten wir nun nochmal unserer TODO-App und setzen diese mittels jQuery um.

Playground — TODO-App (jQuery)

Einige Stellen wurden durch jQuery kürzer. Gerade wenn wir Operationen mit mehreren Elementen durchführen, können wir das mit jQuery kompakter schreiben:

// vorher 
for (const item of todoList.querySelectorAll(".todo-list-item")) {
  item.classList.remove("editing");

// nachher
todoList.find(".todo-list-item").removeClass("editing");

Auch durch Method Chaining konnten wir einige Stellen kürzer schreiben, z.B.:

todoList.find(".toggle").filter(":checked").closest(".todo-list-item").remove()`

Wir sehen aber auch, dass wir manchmal Elemente extra noch einmal in jQuery kappseln müssen ($(e.target)), wo wir vorher direkt auf dem Element arbeiten konnten.

Insgesamt sind die Verbesserungen durch jQuery aber gering. Die Struktur des Codes hat sich nicht geändert. Wir arbeiten immer noch direkt auf den DOM-Elementen und registrieren entsprechende Listener. Die Länge des Codes ist durch Hilfsfunktionen etwas geschrumpft.

Relevanz​

Wie relevant ist jQuery für die heutige Webentwicklung noch? Hier können wir 2 Aspekte betrachten: Statistiken zur Verwendung von jQuery; und technische Aspekte, was jQuery ermöglicht.

Nutzungsdaten​

Schauen wir uns an, ob und wie oft jQuery auf Webseiten eingesetzt wird, stellen wir fest, dass die Bibliothek noch verbreitet ist.

Statistische Daten finden wir etwa bei W3Techs - World Wide Web Technology Surveys, konkret auf der Unterseite Usage statistics and market shares of JavaScript libraries:

jQuery is used by 74.4% of all the websites [that we monitor]

Dies ist bei W3Techs Platz 1 aller JavaScript-Frameworks. Um diese Zahl besser einschätzen zu können, muss aber auch erwähnt werden, dass basierend auf den Daten der gleichen Seite:

WordPress is used by 43.5% of all the websites [that we monitor]

WordPress liefert jQuery standardmäßig mit aus.

Ein weiterer interessanter Datenpunkt ist das 2024 Developer Survey von Stack Overflow. Stack Overflow ist ein beliebte Seite, wo Fragen von Programmierern gestellt und beantwortet werden. In dieser Umfrage wurden Teilnehmer von Stack Overflow gefragt, mit welchen Technologien sie im letzten Jahr gearbeitet haben und im nächsten Jahr arbeiten werden. Hier ist React auf Platz 2 mit knapp 40% und jQuery auf Platz 3 mit 21%.

Bei State of JavaScript - The annual developer survey of the JavaScript ecosystem findet man ähnliche Daten. Das ist eine jährliche Umfrage, an der Entwickler aus der ganzen Welt teilnehmen. Bezüglich jQuery finden wir hier, dass nur 17% der Teilnehmer jQuery regelmäßig verwendet. Bei Front-end frameworks, die wir uns noch anschauen, wurde etwa React von 82% der Teilnehmer schon genutzt, Svelte von 26%.

Um diese Ergebnisse zusammenzufassen: jQuery wird häufig noch im Low-Code-Bereich verwendet, wo kleinere Funktionalitäten und Interaktivität umzusetzen ist. Bei komplexeren und größeren Webapp erhalten andere UI-Frameworks mehr Verwendung. In den nächsten Abschnitten werden wir uns daher einige UI-Frameworks anschauen und welche Probleme diese lösen. Zudem wurden sehr viele Plugins für jQuery erstellt, die Funktionalitäten bereitstellen, für die man sonst erst einmal Alternativen suchen müsste.

Technische Relevanz​

Technisch haben wir bereits oben zu den Bestandteilen von jQuery gesehen, dass viele dieser Teile zwar einmal ihre Berechtigung hatten, inzwischen aber obsolet aufgrund von Alternativen, die der Browser direkt bereitstellt. Zusammenfassend noch einmal:

  • DOM-API-Hilfsfunktionen — Die native DOM API stellt bereits viele nützliche Funktionen bereit. Zudem werden wir sehen, dass die direkte Arbeit mit dem DOM bei größeren Webapps problematisch ist und es andere Ansätze hier gibt.
  • AJAX — Die native fetch-API ist eine moderne, einfach zu verwendende Alternative.
  • JavaScript-Hilfsfunktionen — Inzwischen stellt JavaScript hier entsprechende Funktionen bereit.
  • Deferred — Die nativen Promise erfüllen den gleichen Zweck.
  • Animationen — Durchaus noch relevant, es sollten aber Alternativen wie CSS-Animation verwendet werden, wo möglich.

Cash​

Der auch heute noch relevanteste Bestandteil von jQuery sind die DOM-API-Funktionen.

Cash ist eine sehr kleine JavaScript-Bibliothek (etwa 16KB), welche nur diesen DOM-API-Teil von jQuery enthält. Es stellt daher eine mögliche Alternativ dar, wenn wir direkte Operation auf dem DOM ausführen möchten.
