Abschnitt 4 - Performance (Svelte)

In den letzten beiden Abschnitten wollen wir uns noch 2 Möglichkeiten für die Entwicklung von UIs anschauen, die erst kürzlich entwickelt wurden. Wir beginnen mit Svelte.

info

Aufgrund seiner technischen Struktur ist Svelte in unserer Sandbox aktuell nicht ganz so einfach zu verwenden. Daher sei an dieser Stelle auch auf den offiziellen Svelte Playground verwiesen.

Details

Um Svelte in unserer Sandbox zu verwenden, orientiert sich am besten an den Code-Beispielen unten. Kurz zusammengefasst:

  • Ganz am Anfang des JavaScript muss das magische Wort "svelte"; stehen.
  • Danach müssen in jedem Teil (HTML, CSS, JavaScript) eine oder mehrere Dateien definiert werden, z.B.
// file: Counter.js
// ...

// file: App.js
import Counter from "@sandbox/Counter.js"
// ...

Es muss mindestens die Datei App.js geben, welche den Einstiegspunkt darstellt.

Imports im JavaScript von anderen Dateien müssen dem Format import ... from "@sandbox/filename.js" entsprechen.

Probleme React​

Ein Problem von React, was wir bereits besprochen hatten: durch den Virtual DOM entsteht Overhead, der die Performance beeinflussen kann.

Ein weiteres Problem, was wir noch nicht besprochen hatten: React ist nur für HTML und JavaScript zuständig, nicht für CSS. CSS kann natürlich unabhängig von React geschrieben werden — denn am Ende entsteht bei React auch nur HTML, was wir per CSS stylen können.

Es besteht dann aber die Gefahr, dass verschiedene CSS-Regeln in Konflikte geraten. Wir können versuchen, diese Konflikte zu vermeiden, indem wir etwa Namenskonventionen wie das bereits besprochene BEM verwenden. Wäre es aber nicht schön, wenn ein UI-Framework hier Lösungsansätze bieten könnte?

Svelte​

Svelte ist in vieler Hinsicht ähnlich zu React und Knockout. Wir wollen uns hier daher auf einen kurzen Überblick beschränken.

Grundstruktur​

Im Gegensatz zu React verwendet Svelte separate Teile für HTML, CSS und JavaScript. Ein einfacher Button mit einem Zähler sind in Svelte wie folgt aus:

In Sandbox öffnen
  • HTML
  • CSS
  • JS
svelte-counter
loading...
info

In unserer Sandbox sind HTML, CSS und JavaScript 3 getrennten Tabs. Normalerweise verwendet Svelte Dateien mit der Endung .svelte, diese enthalten HTML, CSS und JavaScript in der folgenden Form:

<script>...</script>
<style>...</style>
<div><button>...</button></div>

Genauso wie bei React sehen wir im HTML, dass Variablen aus dem JavaScript verwendet werden mittels geschweiften Klammern. Das HTML kann der Browser nicht direkt interpretieren, Svelte erzeugt daraus erst HTML für den Browser.

Ebenso wie bei React definieren wir im HTML einen Event-Listener auf das click-Ereignis.

Der Event-Listener selber befindet sich im JavaScript. Im Vergleich zu React sehen wir dort folgendes:

// react: const [count, setCount] = useState(0);
let count = $state(0);

function increment() {
  // react: setState(count + 1);
  count = count + 1;
}

Reaktivität​

Es wird nur eine Variable neu zugewiesen. Bei React informieren das Framework über den Aufruf der setCount-Funktion, dass sich etwas geändert hat? Wie kann Svelte wissen, dass eine Variable neu zugewiesen wurde und daher das HTML aktualisiert werden muss? Die Variable count ist auch kein speziellen Objekt, was wir prüfen, können in dem wir uns den Wert per console.log(typeof count) ausgeben: es ist einfach nur eine number.

Die Antwort liegt im Svelte-Compiler. Jede *.svelte-Datei wird von Svelte analysiert und entsprechend vorbereitet. Mittels count = $state(0) informieren wir Svelte, dass count eine reaktive Variable ist, deren Wert Auswirkung auf die UI hat.

Ebenso analysiert Svelte auch das HTML und prüft, wo die Variable count verwendet wird. Durch dieses Vorgehen ist Svelte in der Lage, im Voraus zu bestimmen, welche Teile des HTML angepasst werden müssen, wenn sich die Variable count ändert (nämlich der textContent des Buttons).

In anderen Worten: Svelte benötigt keinen Virtual DOM mehr! Es kann bei Änderungen direkt den DOM im Browser aktualisieren und muss keine Differenz zwischen dem Virtual DOM und dem Browser-DOM mehr berechnen.

Bei Interesse ist hier das JavaScript der Counter.js aus obigen Beispiel, nachdem der Svelte-Compiler es analysiert und umgewandelt hat:

Details
import 'svelte/internal/disclose-version';
import * as $ from 'svelte/internal/client';

function increment(_, count) {
	$.set(count, $.get(count) + 1);
}

var root = $.template(`<button class="svelte-1l68e0"> </button>`);

export default function Counter($$anchor) {
	let count = $.state(0);
	var button = root();

	button.__click = [increment, count];

	var text = $.child(button);

	$.reset(button);

	$.template_effect(() => $.set_text(text, `Clicked ${$.get(count) ?? ''}
	${($.get(count) === 1 ? 'time' : 'times') ?? ''}`));

	$.append($$anchor, button);
}

$.delegate(['click']);

CSS-Scoping​

Ein weiterer angenehmer Nebeneffekt, den wir aus der Tatsache erhalten, dass Svelte die *.svelte-Dateien genauer analysiert: Scoping von CSS.

Svelte stellt sicher, dass unser CSS nur Auswirkung auf die aktuelle Komponente haben kann, für die es gedacht war. Das erreicht Svelte auf einfache Weise. Schauen wir nochmal in das CSS und JavaScript, welches der Svelte-Compiler erzeugt:

var root = $.template(`<button class="svelte-1l68e0"> </button>`);
button.svelte-1l68e0 {
  background-color: #333;
  color: white;
  border: none;
  padding: 0.5em 2em;
  border-radius: 10px;
}

Svelte generiert einfach unvorhersehbare CSS-Klassen und schreibt diese mit an das HTML und CSS. Auf diese Weise ist sichergestellt, dass die entsprechende CSS-Regel nicht auf andere Elemente im DOM Auswirkung hat.

Ergebnisse​

Bevor wir uns Svelte im Vergleich anschauen, hier unser TODO-App-Beispiel mit Svelte:

Playground — TODO-App (Svelte)

Vergleichen wir die Lösung in Svelte mit React, fällt noch einmal auf, wie sehr die beiden UI-Frameworks sich ähneln, beispielsweise bei useState / $state und useEffect / $effect.

Die klare Trennung von Svelte zwischen HTML und JavaScript hat mir persönlich besser gefallen. Das mag subjektiv sein, war aber mein Eindruck beim schreiben der TODO-App. In React werden Komponenten schneller unübersichtlich, wenn diese sowohl HTML als auch JavaScript vermischen.

Bezüglich der Übersicht des Codes selber erzwingt Svelte auch die strikte Trennung von Komponenten: jede Datei kann nur eine einzelne Komponente enthalten. Während es in React zwar auch empfehlenswert ist, verschiedene Dateien für verschiedene Komponenten zu verwenden, ist das nur eine Empfehlung. In einem großen Team kann es so eher passieren, dass der Code unübersichtlich wird.

Ein weiterer Unterschied bezüglich der Lesbarkeit besteht darin, dass bei Svelte das Aktualisieren der Daten einfacher ist. Wollen wir etwa ein neues Element zu einem Array hinzufügen, müssen wir bei React immer neues Array erzeugen, sonst erkennt React die Änderung nicht.

setItems([...items, {id: "10", name: "Item 10"}]);

Bei Svelte können wir auch ein neues Array erzeugen, wenn wir möchten. Es ist aber auch nicht falsch, ein neues Element direkt in das Array zu pushen.

items.push({id: "10", name: "Item 10"});

Ebenso können wir in Svelte auch direkt ein Element im Array modifizieren, was bei React nicht erlaubt ist.

items[2].name = "New name";

Durch diese Möglichkeiten sind uns keine Grenzen gesetzt beim Schreiben von Code, sondern wir können Änderungen an den Daten so implementieren, wie wir es am einfachsten und am leichtesten zu lesen finden. Zudem reduziert das auch Fehler — es ist schwerer, Code zu schreiben, der nicht funktionieren wird.

info

Technisch verwendet Svelte derzeit sogenannte Proxy, um zu überwachen, wie und wann ein Objekt oder Array sich verändert.

Eine weiterer Vorteil für die Lesbarkeit des Codes bietet Svelte bei bedingten und wiederholten Elementen. Erinnern wir zurück an React, hier haben wir reine JavaScript-Konstrukte verwendet.

{ items.length > 0 ? <TodoFooter /> : "Keine TODOs"}
{ items.map(item => <TodoItem key={item.id} item={item} />) }

Bei Svelte steht uns separate Syntax zur Verfügung, die mehr an if-else und for ... of ... erinnert.

{#if items.length > 0}
  <TodoFooter />
{:else}
  "Keine TODOs"
{/if}

{#each items as item (item.id)}
  <TodoItem item={item}/>
{/each}

Das ist etwas länger als einfache Fälle bei React, aus meiner Sicht aber klarer strukturierter und einfach zu schreiben und zu lesen.

Noch ein weitere Vorteil besteht in der Behandlung der Abhängigkeiten. In React ist es erforderlich, etwa beim useEffect-Hook manuell anzugeben, von welchen Werten dieser abhängt. Ändern sich einer dieser Werte, wird der Effekt erneut ausgeführt. Aufgrund der Tatsache, dass Svelte die *.svelte-Dateien mit den Komponenten einer gründlicheren Analyse unterzieht, ist das bei Svelte nicht erforderlich. Svelte erkennt automatisch, von welchen Werte ein $effect abhängt.

Schließlich verbessert Svelte auch die Performance, indem es wie besprochen keinen Virtual DOM mehr benötigt, sondern Änderungen direkt auf dem DOM ausführt. Hier sei aber nochmal ausdrücklich darauf hingewiesen, dass Performance immer praktisch getestet werden muss. Abhängig vom Anwendungsfall kann der Unterschied in der Performance auch gering oder nicht vorhanden sein. js-framework-benchmark ist ein Benchmark, wo Svelte etwas besser als React abschneidet. Der dort getestete Testfall muss aber nicht zwingend Aussage über echte Fälle in der Praxis liefern.

Bezüglich der Performance sollten wir noch bedenken, dass bei Svelte weniger manuelle Optimierungen notwendig sind. Bei React gibt es verschiedene Hooks wie useMemo oder useCallback, welche im Wesentlichen nur dafür gedacht sind, Probleme mit dem Render-Modell und Virtual DOM zu beheben. Als Entwickler müssen wir diese aber manuell und mit Bedacht einsetzen.
